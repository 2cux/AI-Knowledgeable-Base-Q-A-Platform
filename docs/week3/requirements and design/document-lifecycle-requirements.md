# 第三周文档生命周期需求

## 1. 背景与当前现状

### 1.1 当前已完成能力

当前文档链路已经完成：

- 文档元数据上传。
- 真实 `txt/md` 文件上传。
- 本地存储。
- 文档解析切片。
- 切片入库。
- 文档 embedding。
- 基于文档内容检索和问答。

### 1.2 当前不足

当前文档能力更偏“接入闭环”，缺少真正的生命周期治理：

- 没有文档删除能力。
- 没有文档替换更新能力。
- 没有正式的重新处理 / 重建向量闭环。
- 文档状态可见性不足。
- 文档失败后的重试治理不清晰。
- 没有版本管理基础能力。
- 没有发布态 / 失效态治理。

## 2. 第三周目标

第三周文档生命周期目标是把文档从“能接进来”推进到“能维护、能重建、能治理”：

- 提供基础删除能力。
- 提供重新处理和重新 embedding 的明确路径。
- 补强状态可见性和失败可追踪性。
- 为替换更新和版本基础能力建立可落地方案。
- 保证 chunk / embedding / retrieval 之间不产生明显脏数据。

## 3. 范围边界

### 3.1 第三周必须做

- 文档列表 / 详情增强。
- 文档删除。
- 重新处理。
- 重新 embedding。
- 状态可见性增强。
- 失败重试基础能力。

### 3.2 第三周建议做

- 文档替换更新基础版。
- 最小版本管理语义。

### 3.3 第三周不做

- 不做完整文档审核流。
- 不做复杂文档发布平台。
- 不做复杂版本 diff 视图。
- 不做对象存储迁移。

## 4. 需求点

## 4.1 文档列表增强

### 需求说明

当前列表能查基础元数据，但不足以支撑运维与治理。

### 第三周要求

文档列表建议增强以下字段：

- `parseStatus`
- `chunkCount`
- `embeddingStatus`
- `latestTaskStatus`
- `latestErrorMessage`
- `updatedAt`

### 目标

- 用户和管理员能快速判断文档当前是否可检索、是否失败、是否需要重试。

## 4.2 文档详情增强

### 第三周要求

详情建议补充：

- 解析状态。
- 切片数量。
- embedding 汇总状态。
- 最近任务状态。
- 最近错误信息。
- 是否支持重新处理。
- 是否支持重新 embedding。

## 4.3 文档删除

### 需求说明

当前文档只能上传，不能删除，会导致知识库内容不可治理。

### 第三周要求

- 用户可以删除自己知识库下的文档。
- 删除时要明确处理以下数据：
- 文档元数据。
- `document_chunk`
- `chunk_embedding`
- 相关本地文件。

### 完成标准

- 删除后文档不再出现在列表。
- 已删除文档不会继续被检索到。
- 不留下明显脏数据。

## 4.4 文档重新处理

### 需求说明

当前已 `CHUNKED` 的文档会被直接拦截，无法正式重建切片。

### 第三周要求

- 支持文档重新解析切片。
- 明确 reprocess 与普通 process 的差异。
- 重建前需清理旧 chunk 及其相关 embedding 状态。

### 范围说明

第三周可采用简单模式：

- 显式 `force=true` 或独立 reprocess 接口。

## 4.5 文档重新 embedding

### 需求说明

当前向量化可以重跑单文档流程，但缺少正式产品语义和状态治理。

### 第三周要求

- 支持显式重新 embedding。
- 支持指定模型名重建。
- 明确旧向量状态如何覆盖或作废。

## 4.6 状态可见性

### 需求说明

当前 `document.parse_status`、`task_record.status`、`chunk_embedding.status` 分散存在，但对外状态聚合仍偏弱。

### 第三周要求

- 提供文档级统一状态视图。
- 让用户清楚看到：
- 是否已上传。
- 是否已切片。
- 是否可检索。
- 是否有失败。
- 最近失败原因是什么。

## 4.7 失败重试

### 需求说明

文档处理和 embedding 是典型容易失败的后台动作，第三周需要最小重试语义。

### 第三周要求

- 失败后允许用户手动重试。
- 重试时必须保证旧失败状态不会污染新结果。
- 重试动作要可追踪。

## 4.8 文档替换更新

### 需求说明

真实知识内容会变化，仅有删除再上传体验较差。

### 第三周要求

- 提供基础替换上传方案。
- 替换后应重新触发解析和向量化。
- 替换前旧数据要进入失效或被覆盖逻辑。

### 范围边界

第三周只做基础版，不要求复杂版本树和版本对比。

## 4.9 版本管理基础版

### 第三周目标

建立最小版本语义，而不是完整文档版本平台。

### 最小要求

- 当前有效版本可识别。
- 替换后旧版本不再参与检索。
- 至少能保留“原文档被替换过”的事实。

## 5. P0 / P1 优先级

### P0

- 文档删除。
- 重新处理。
- 重新 embedding。
- 状态可见性增强。
- 失败重试基础能力。

### P1

- 文档替换更新基础版。
- 最小版本管理语义。

## 6. 与现有实现的兼容策略

- 保留现有 `upload-file`、`process`、`embed` 主接口。
- 在现有模型基础上增量补字段或补接口，不推翻当前 `document/document_chunk/chunk_embedding/task_record` 结构。
- 老文档数据继续可读。

## 7. 风险与注意事项

- 删除和重建动作容易产生脏数据，需要优先保证一致性。
- 文档生命周期能力会直接影响检索质量和问答稳定性，不能只做接口表面。
- 替换更新如果没有旧数据失效策略，可能导致同一知识重复召回。

## 8. 验收标准

- 文档可被安全删除，删除后不再参与检索。
- 文档可明确重新处理和重新 embedding。
- 文档列表与详情能看出真实处理状态。
- 失败后能手动重试，并且状态可追踪。
- 如实现替换更新，旧版本不会继续被检索使用。

## 9. 与旧文档差异说明

第二周文档主要聚焦“真实文档进入系统并跑通 RAG 主链路”。

第三周把重点切换到“文档可维护、可重建、可治理”，这是从 Alpha 走向更稳定阶段必须补齐的能力。
