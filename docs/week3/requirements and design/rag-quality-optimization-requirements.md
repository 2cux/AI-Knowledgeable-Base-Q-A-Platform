# 第三周 RAG 质量优化需求

## 1. 背景与当前现状

### 1.1 当前已完成能力

当前项目已经具备基础 RAG 主链路：

- 用户可以在自己的知识库内发起提问。
- 系统会先做检索，再做答案生成。
- 当前检索结果会转换为 `citations` 返回给 `chat/ask`。
- 问答结果会落到 `chat_record`。

### 1.2 当前真实问题

当前 RAG 质量问题主要集中在以下几个方面：

- 检索仍是基础版，基于 `chunk_embedding.vector_json` 存储向量，由 Java 应用层做余弦相似度计算。
- 命中策略较简单，仅靠固定阈值筛选，缺少更系统的召回与拒答策略。
- 真实 LLM 已接入，但答案输出治理仍不够完整，存在答案抽取和结果格式不稳定问题。
- 用户详情和管理详情中的问答追溯不完整，`citations/topK/conversationId` 没有完整暴露。
- 文档外问题虽然有基础拒答文案，但仍不够成熟，缺少清晰判定规则。
- 当前测试更多依赖手工 / Apifox 验收，缺少质量评估集与可重复校验基线。

## 2. 第三周目标

第三周在 RAG 质量优化上的目标不是追求“高级算法堆叠”，而是建立**最小可控质量闭环**：

- 提升检索命中判定的可信度。
- 提高文档外问题的拒答准确性。
- 提高答案输出结果的稳定性和可用性。
- 补齐问答详情的追溯能力。
- 为后续 rerank、检索评估、Prompt 配置化预留清晰接口。

## 3. 范围边界

### 3.1 第三周必须覆盖

- 检索质量基础治理。
- 拒答与命中策略收敛。
- 答案输出治理。
- 引用追溯补齐。
- 质量评估样例与验收标准。

### 3.2 第三周不做

- 不强制接入正式向量数据库。
- 不强制接入真实 rerank 服务。
- 不做复杂模型路由系统。
- 不做自动化 Prompt 平台。
- 不做复杂在线评估平台。

## 4. 需求点

## 4.1 检索质量基础治理

### 需求说明

当前检索能用，但质量策略过于单一。第三周需要建立最小可配置、可解释的检索命中规则。

### 具体要求

- 明确“有效命中 chunk”的最低判定条件。
- `topK` 继续保留，但要区分“召回数量”和“有效引用数量”。
- 对低分 chunk 进行过滤，避免把弱相关内容塞给 LLM。
- 保留当前“DB 存向量 + Java 算相似度”的实现路径，但文档中必须明确这只是基础版能力。

### 完成标准

- 检索服务中命中规则不再只停留在代码硬编码语义，而要在设计文档和验收标准中明确。
- 对于同样的测试问题，检索结果具备稳定的有效返回策略。

## 4.2 拒答策略优化

### 需求说明

当前无有效 chunk 时可以拒答，但“边缘命中”“弱命中”“命中数量少但相关性低”场景仍缺少清晰策略。

### 具体要求

- 明确以下场景的处理规则：
- 完全无命中。
- 有召回但低于有效阈值。
- 有少量命中但不足以支持回答。
- LLM 服务调用失败。

- 区分：
- 检索未命中导致的拒答。
- 模型失败导致的降级提示。
- 有命中但证据不足导致的保守回答。

### 完成标准

- `matched` 的含义必须稳定，反映“是否有足够可信的检索命中”，不能只反映“LLM 是否返回了文本”。
- 对外响应中的拒答文案和内部记录逻辑保持一致。

## 4.3 回答格式治理

### 需求说明

当前真实 LLM 已接入，但答案输出仍偏粗糙。第三周需要保证用户拿到的是“可展示、可理解、可落库”的答案文本，而不是不稳定的上游原始结构。

### 具体要求

- 明确答案抽取规则。
- 明确空回答、异常回答、无结构回答的兜底策略。
- 规范回答文本风格：
- 简洁。
- 中文优先。
- 不夸大、不编造。
- 与检索证据一致。

- 建立“最终落库答案”和“对外返回答案”的一致规则。

### 完成标准

- `chat/ask` 对外返回的答案是稳定文本，不依赖上游原始 JSON 直接透传。
- `chat_record.answer` 存储的是最终可展示答案。

## 4.4 引用追溯补齐

### 需求说明

当前 `citations` 只在 `chat/ask` 响应中较完整，问答详情和管理详情追溯能力不足。

### 具体要求

- 用户问答详情补齐：
- `conversationId`
- `topK`
- `citations`

- 管理端问答详情补齐：
- `userId`
- `conversationId`
- `topK`
- `citations`

- 引用结构需可直接映射到：
- `documentId`
- `chunkId`
- `chunkIndex`
- `documentName`
- `score`
- `contentSnippet`

### 完成标准

- 用户详情与管理详情可完整追溯到本次回答所依据的知识片段。

## 4.5 Prompt 治理最小规范

### 需求说明

第三周不做复杂 Prompt 平台，但必须明确 Prompt 的最小治理规则。

### 具体要求

- 区分 system prompt 和 user prompt。
- 明确 history、chunk、question 的拼装顺序。
- 明确拒答要求必须写入 system prompt。
- 明确历史上下文长度控制规则。

### 完成标准

- Prompt 结构在文档中有清晰定义。
- 后续即使做 Prompt 配置化，也不会推翻第三周结构。

## 4.6 质量评估集与验收基线

### 需求说明

当前主要依赖手工 Apifox 验收，无法稳定判断 RAG 质量是否提升。

### 具体要求

- 为第三周建立最小评估集。
- 评估集至少覆盖：
- 命中类问题。
- 边缘命中问题。
- 未命中问题。
- 多轮追问问题。
- 文档外问题。

- 每类问题需要定义：
- 测试文档。
- 输入问题。
- 预期检索表现。
- 预期回答表现。

### 完成标准

- 第三周结束时，RAG 质量优化必须有一组固定样例可重复验收。

## 5. 与现有实现的兼容策略

- 保留当前 `chat/ask` 主接口，不新开平行问答主链路。
- 保留当前检索接口和 `topK` 语义，但增强有效命中判定。
- 保留当前 `chat_record.citations_json` 字段，不引入独立引用表。
- 保留当前 `conversationId` 输入输出，后续与会话系统演进兼容。

## 6. 风险与注意事项

- 如果直接引入过复杂的 rerank 或模型编排，会超出当前项目节奏。
- 质量优化必须建立在当前真实代码基础上，不能假设已有正式向量库。
- 对 `matched`、`topK`、`citations` 等字段语义的调整，需要保证旧数据兼容。

## 7. 验收标准

- 命中问题的检索结果更稳定，返回的 `citations` 与答案一致。
- 文档外问题能够稳定拒答，不再轻易给出似是而非答案。
- `chat/ask` 输出为稳定文本，不直接暴露上游原始 JSON。
- 用户详情与管理详情的追溯字段补齐。
- 第三周形成最小固定评估集，并可重复执行。

## 8. 与旧文档的差异说明

第二周文档对 RAG 质量的关注点更偏“打通真实链路”。

第三周开始，重点从“能调通”转向“质量更稳、追溯更完整、拒答更可控”。因此本周文档会显式强化以下内容：

- 当前检索不是正式版向量基础设施。
- 当前拒答与命中语义需要重新收敛。
- 当前详情追溯能力不完整，必须补齐。
