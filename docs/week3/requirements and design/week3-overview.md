# 第三周总览

## 1. 文档定位

本文档用于定义第三周的整体定位、目标边界和交付物总览，作为第三周需求与设计文档体系的入口。

第三周不是重新定义项目方向，也不是继续横向堆功能模块，而是基于当前项目已经达到的真实状态，推动系统从“Alpha 主链路可用”向“准 Beta 可规划、质量更稳、维护成本更低”演进。

## 2. 当前项目所处阶段

### 2.1 真实阶段判断

当前项目处于：**Alpha 可用期（后端真实主链路已打通）**

### 2.2 判断依据

基于当前真实代码、第二周文档和已有联调结果，项目已经具备以下能力：

- 用户注册、登录、JWT 基础鉴权。
- 知识库创建、列表、详情。
- 真实文件上传，当前真实闭环支持 `txt/md`。
- 本地受控目录存储。
- 基于真实文件的解析、切片、切片入库。
- 文档向量化状态表与真实 embedding 调用。
- 基于 `chunk_embedding.vector_json` 的基础检索链路。
- 基于检索结果的基础 LLM 问答。
- `citations` 在问答响应中返回，并写入 `chat_record`。
- 问答记录、用户反馈、管理端日志、未命中问题查询。
- 统一异常处理、统一响应结构、Flyway 迁移。

### 2.3 当前仍然明显不足的部分

- 检索仍是“**DB 存向量 + Java 算相似度**”的基础版，不是正式版向量检索基础设施。
- `conversationId` 目前只能算轻量串联，不是完整的会话系统。
- 问答详情追溯不完整，用户详情和管理详情未完整返回 `citations/topK/conversationId/userId` 等信息。
- 文档生命周期不完整，缺少删除、替换、重新处理、版本与发布治理。
- 权限体系仍是 owner 级基础隔离，不是正式版级别。
- 存在高优先级安全风险：明文 API Key、`/debug/ai/**` 匿名放行。
- 测试仍以 Apifox 手工验收为主，没有工程化自动化测试体系。

## 3. 第三周核心目标

第三周的核心目标不是“功能数量增加”，而是：

- 提升 RAG 回答质量与链路稳定性。
- 把轻量会话串联升级为最小可用会话系统。
- 补齐文档生命周期管理的最小闭环。
- 把管理端从“日志查看”推进到“基础运营治理”。
- 收口安全风险并加强工程化基础。

换句话说，第三周的目标是把当前 Alpha 后端从“主链路能跑”推进到“边界更清晰、数据更可追、能力更可维护、后续 Beta 演进成本更低”。

## 4. 第三周不做什么

第三周明确不做以下内容，避免范围失控：

- 不做复杂多租户体系。
- 不做完整细粒度 RBAC 与部门组织权限模型。
- 不做外部知识源对接，如 Wiki、CRM、工单、OA、IM。
- 不做复杂 Agent、Tool Calling、工作流编排。
- 不做前端页面大规模重构。
- 不做微服务拆分、分布式调度、高可用架构升级。
- 不做正式版向量数据库重平台迁移。
- 不做复杂报表平台和 BI 看板。

第三周优先“增量演进”，不推翻当前 Spring Boot 单体后端结构。

## 6. 第三周文档体系说明

### 6.1 文档组织原则

- 以当前真实代码和真实进度为准。
- 以第三周可直接开发为目标。
- 需求文档说明“为什么做、做到什么程度、如何验收”。
- 设计文档说明“如何增量落地、影响哪些模型/API/状态流转”。

### 6.2 与第二周文档的关系

第二周文档重点是把主链路从骨架推进到真实可演示闭环。

第三周文档重点不是重复第二周链路建设，而是针对第二周完成后的真实短板做补齐和收敛，尤其是：

- 从“能回答”推进到“回答质量可控”。
- 从“有 conversationId”推进到“最小可用会话系统”。
- 从“能上传并处理文档”推进到“文档可维护、可重建、可治理”。
- 从“有日志可看”推进到“有基础运营治理能力”。
- 从“可开发演示”推进到“可更安全地继续迭代”。

### 6.3 与旧需求文档的差异说明

原始需求文档中存在较多正式版目标，例如复杂权限、知识库共享、审核流、内容发布、模型与 Prompt 配置、多来源接入等。

第三周不会一次性实现这些正式版目标，而是按当前真实阶段进行收敛：

- 优先做真正影响 Alpha 到 Beta 过渡的能力。
- 对正式版能力仅做必要预留，不做超前复杂设计。
- 对已有旧文档中未落地的能力，文档中会明确标记为“延后”或“仅做预留”。

## 7. 第三周完成标准

第三周完成后，项目应至少达到以下状态：

- 检索、拒答、引用追溯、回答格式治理具备明确规则和最小实现方案。
- 会话列表、会话详情、上下文加载策略有清晰模型与接口设计，并完成最小增量演进。
- 文档生命周期具备删除、重新处理、重新 embedding、状态可见性的基础能力。
- 管理端日志、未命中治理、反馈运营具备基础闭环。
- 高风险 debug 接口、明文密钥、测试与配置隔离问题进入明确治理状态。

## 8. 风险提示

- 当前代码与第二周部分规划文档并不完全一致，例如文档中规划的 `parse/run`、问答详情增强、会话模型闭环等未全部落地。
- 第三周文档必须继续以当前真实实现为起点，不能直接按旧规划假设“已经完成”。
- 如果第三周仍继续横向扩张功能面，会直接稀释当前最关键的质量和工程问题。
