# 第三周范围与优先级

## 1. 文档定位

本文档用于定义第三周的范围收敛策略、优先级分层和明确延后项，确保第三周开发围绕当前项目真实短板展开，而不是偏离主线。

## 2. 当前现状与收敛动机

### 2.1 当前已完成能力

当前项目已经完成 Alpha 后端主链路：

- 认证、知识库基础能力。
- 真实 `txt/md` 上传、本地存储、解析、切片。
- 文档处理与向量化主链路。
- 真实 embedding。
- 基础检索与基础 LLM 问答。
- 问答记录、反馈、管理端基础查询。
- 第二周端到端真实主链路验收文档和手工联调结果。

### 2.2 当前最突出问题

- RAG 回答质量和可解释性仍偏基础版。
- 会话能力只有 `conversationId` 轻量串联。
- 文档只具备接入能力，不具备维护能力。
- 管理端只有“看日志”，没有“治理闭环”。
- 安全与工程化存在较明显风险。

### 2.3 为什么第三周必须收敛范围

如果第三周继续横向增加新模块：

- 已有主链路质量问题会继续累积。
- 产品看起来更“全”，但真实可维护性和可上线程度不会提升。
- 后续进入 Beta 时，数据模型、接口契约和治理逻辑的返工成本会更高。

因此第三周必须聚焦在“提升已有主链路质量与边界清晰度”。

## 3. 第三周总体优先级策略

第三周按 `P0 / P1 / P2` 划分。

### 3.1 P0 定义

P0 表示：第三周必须完成，否则当前 Alpha 后端难以进入准 Beta 规划阶段。

### 3.2 P1 定义

P1 表示：强烈建议完成，能显著提升系统完整性与可维护性，但不应阻塞 P0。

### 3.3 P2 定义

P2 表示：时间允许再做，适合做预留或局部增强，不应抢占 P0/P1 资源。

## 4. 第三周必须做的内容

## 4.1 P0 列表

### P0-1 RAG 质量优化基础闭环

- 检索结果阈值与命中策略明确化。
- 拒答策略收敛，避免“检索弱命中也强行回答”。
- 问答答案输出抽取与格式治理。
- `citations` 详情追溯补齐。
- 基础质量评估集与验收口径建立。

### P0-2 会话系统最小可用演进

- 从仅有 `conversationId` 升级到最小可用 `conversation/message` 模型。
- 支持会话列表、会话详情、消息视图。
- 明确与现有 `chat_record` 的关系与兼容方案。

### P0-3 文档生命周期基础闭环

- 文档删除能力。
- 文档重新处理 / 重新 embedding。
- 文档状态可见性增强。
- 文档失败重试基础能力。
- 避免 chunk / embedding / retrieval 脏数据残留。

### P0-4 工程化与安全高风险收口

- 收口匿名 debug AI 接口。
- 移除或隔离明文 API Key 风险。
- 建立第三周最小回归测试与验收基线。
- 规范错误语义、日志和配置隔离。

## 4.2 P1 列表

### P1-1 文档更新替换与基础版本能力

- 支持基础替换上传。
- 建立最小版本语义，而不是完整版本平台。

### P1-2 管理端运营增强

- 问答日志详情增强。
- 未命中问题分类治理基础方案。
- 反馈运营查看能力。
- 基础热门问题 / 基础统计能力。

### P1-3 Prompt/模型配置预留

- 不强制做完整配置后台。
- 但应明确配置模型、Prompt 模板的后端结构预留方案。

## 4.3 P2 列表

### P2-1 检索质量增强预留

- rerank 适配层预留。
- 更正式的评估指标框架预留。
- 检索策略配置化增强。

### P2-2 更复杂权限体系预留

- RBAC 与数据范围模型方向。
- 文档级权限、成员共享的演进路线。

### P2-3 可观测性增强

- 监控、审计、告警、异步队列平台化方案预留。

## 5. 明确延后的内容

以下内容第三周明确延后：

- 多租户体系。
- 知识库共享与复杂成员协作。
- 正式版文档审核流、发布流、审批流。
- 多知识库智能路由。
- 外部知识源集成。
- 复杂 BI 看板。
- 微服务拆分与分布式任务平台。
- 完整 Agent / Tool Calling 编排。

延后原因：

- 与第三周“质量优化与边界补齐”的主目标不一致。
- 当前代码基础尚未达到适合承接这些复杂能力的成熟度。

## 6. 为什么这些优先级最合理

### 6.1 对产品价值最直接

第三周用户最能感知的问题，不是“缺少新菜单”，而是：

- 回答质量稳不稳。
- 历史会话能不能真正查看和复用。
- 文档出错后能不能重建和维护。
- 管理端能不能定位问题。

### 6.2 对后续开发成本最关键

如果不先解决：

- 数据模型会继续散落在 `chat_record`、`chunk_embedding` 和 `document` 周边。
- 第四周以后再加功能时，会放大耦合和返工。

### 6.3 对上线风险最敏感

当前匿名 debug AI、明文 API Key、缺少自动化测试，是比“缺少某个业务功能”更紧迫的风险。

## 7. 与最终版目标的关系

第三周不是实现最终版，而是为最终版铺设更可靠的演进路径。

对应关系如下：

- 最终版要求高质量 RAG。
当前第三周先做：阈值、拒答、追溯、格式治理、评估集。

- 最终版要求完整会话系统。
当前第三周先做：最小可用会话列表、会话详情、上下文治理。

- 最终版要求完整知识运营与发布。
当前第三周先做：文档生命周期基础闭环与管理端基础治理。

- 最终版要求正式权限与工程化能力。
当前第三周先做：高风险安全收口与最小回归基线。

## 8. 第三周完成标准

第三周的范围完成，不以“新增接口数量”衡量，而以以下结果衡量：

- RAG 质量优化项形成可开发、可验收闭环。
- 会话系统不再只是 `conversationId` 轻量串联。
- 文档从“能接入”进化到“能维护”。
- 管理端具备基础运营治理能力。
- 高风险安全项和工程项被明确收口。

## 9. 风险与注意事项

- 第三周内容仍需坚持增量改造，避免重构冲动。
- 会话模型与文档生命周期变更都可能影响现有接口，必须优先保证兼容。
- 如果时间不足，应优先保 P0，不要为了做“看起来更完整”的 P1/P2 而压缩 P0。
