# 第三周管理端与运营能力需求

## 1. 背景与当前现状

### 1.1 当前已完成能力

当前管理端相关能力包括：

- 管理端问答日志列表。
- 管理端问答日志详情基础版。
- 未命中问题列表。
- 用户反馈提交通道。

### 1.2 当前不足

虽然已有基础查询，但还不能算完整运营闭环，主要问题包括：

- 日志详情字段不完整，排障信息不足。
- 未命中问题只有查看，没有治理。
- 反馈只是被提交，没有运营视角查看和处理。
- 没有基础热门问题分析。
- 没有模型/Prompt 配置后台，甚至没有明确后端配置预留结构。

## 2. 第三周目标

第三周管理端与运营能力的目标是：

- 把管理端从“日志查看”推进到“基础治理与排障”。
- 提供后端接口和数据能力，先不依赖复杂前端页面。
- 明确哪些能力第三周必须落地，哪些仅做后端结构预留。

## 3. 范围边界

### 3.1 第三周必须做

- 问答日志详情增强。
- 未命中问题治理基础版。
- 反馈运营查看基础版。
- 基础热门问题统计。

### 3.2 第三周建议做

- 基础 Prompt / 模型配置后端预留。
- 基础数据统计接口。

### 3.3 第三周不做

- 不做复杂可视化大屏。
- 不做复杂 BI 聚合平台。
- 不做复杂运营工作流。
- 不做多角色后台菜单体系。

## 4. 需求点

## 4.1 问答日志增强

### 当前问题

当前日志详情不足以支撑有效排障，缺少关键追溯字段。

### 第三周需求

管理端日志详情至少补齐：

- `userId`
- `knowledgeBaseId`
- `conversationId`
- `question`
- `answer`
- `matched`
- `retrievedChunkCount`
- `topK`
- `citations`
- `createdAt`

### 目标

- 让运营或技术人员看到一条问答时，能够直接判断：
- 问的是谁。
- 在哪个知识库。
- 命中了什么片段。
- 为什么命中或未命中。

## 4.2 未命中问题治理

### 当前问题

当前未命中问题只依赖 `matched=false` 记录列表，缺少治理语义。

### 第三周需求

未命中治理基础版需要支持：

- 查看未命中问题列表。
- 按知识库筛选。
- 按时间范围筛选。
- 至少预留“处理状态”或“治理状态”的设计方向。

### 第三周范围说明

第三周可以先不新增独立 `missed_question` 表，但要明确：

- 继续复用 `chat_record` 的方案能支撑到什么程度。
- 如果要进入下一阶段治理，何时需要独立治理表。

## 4.3 反馈运营

### 当前问题

当前用户可以提交反馈，但后台无法查看和利用反馈。

### 第三周需求

需要新增基础反馈运营能力：

- 反馈列表查询。
- 支持按知识库、时间、反馈类型过滤。
- 能看到：
- 对应问答记录。
- 提交用户。
- 反馈类型。
- 备注。
- 创建时间。

### 目标

- 让反馈从“被记录”升级为“可运营分析”。

## 4.4 热门问题分析

### 当前问题

当前没有热门问题、重复问题、访问频次的基础统计。

### 第三周需求

第三周只做基础统计，不做复杂前端看板：

- 按知识库统计高频问题。
- 按时间范围统计问题出现次数。
- 至少支持查看 TopN 问题。

### 范围说明

可基于 `chat_record.question` 做基础聚合，不要求做复杂语义去重。

## 4.5 基础数据统计

### 建议统计口径

- 问答总量。
- 命中量 / 未命中量。
- 反馈总量。
- LIKE / DISLIKE 数量。
- 知识库维度问答量。

第三周重点是形成基础后端统计接口，不要求复杂报表展示。

## 4.6 Prompt / 模型配置预留

### 当前问题

当前 Prompt 写在代码里，模型配置散落在配置文件，运营不可见。

### 第三周需求

第三周不强制做完整配置后台，但至少需要在文档和设计中明确：

- 后续 Prompt 配置如何抽象。
- 后续模型配置如何抽象。
- 第三周是否需要只读配置接口或管理预留结构。

### 第三周建议

- 可先提供只读型能力说明。
- 或仅做后端结构预留，不要求管理界面。

## 5. 与现有实现的兼容要求

- 管理端能力继续以当前 `/api/admin/**` 结构为基础扩展。
- 不影响普通用户端主链路。
- 不推翻当前 `chat_record`、`chat_feedback` 的使用方式。

## 6. 风险与注意事项

- 如果第三周直接引入过重运营模型，会偏离当前项目阶段。
- 热门问题聚合若不做清晰范围定义，容易陷入复杂 NLP 去重问题。
- 未命中问题治理要注意与后续独立问题池之间的演进兼容。

## 7. 验收标准

- 管理端问答详情可支撑基本排障。
- 能查看并筛选用户反馈。
- 能查看基础未命中问题并形成治理入口。
- 能查看基础热门问题 / 基础统计数据。
- Prompt/模型配置至少形成后续可落地的后端演进方案。

## 8. 与旧文档差异说明

原需求文档中的管理端目标较宽，包括权限管理、内容审核、模型配置、统计分析等完整后台能力。

第三周不会直接实现完整后台，而是聚焦当前项目最需要的运营支撑：

- 问答可排障。
- 未命中可治理。
- 反馈可查看。
- 统计有基础。
