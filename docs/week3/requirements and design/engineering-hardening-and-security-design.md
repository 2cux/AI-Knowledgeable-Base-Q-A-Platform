# 第三周工程化加固与安全收口设计

## 1. 设计背景

当前项目已经达到 Alpha 后端主链路可用状态，但工程和安全层面仍有明显短板：

- 配置文件中存在明文 API Key 风险。
- `/debug/ai/**` 调试接口在当前实现中匿名放行，且默认启用。
- 测试仍以手工 / Apifox 为主，没有系统化自动回归。
- 错误语义、日志、配置隔离、任务治理仍偏基础。

第三周必须对这些问题做收口，否则系统的可演示性和可维护性会被安全与工程风险拖住。

## 2. 设计目标

- 收口高风险调试接口。
- 收口敏感配置与环境隔离风险。
- 明确异常码和错误语义规范。
- 增强日志、审计和任务排障能力。
- 建立最小自动化测试与回归基线。
- 对异步任务演进给出清晰路线，但不做过度架构。

## 3. 当前问题拆解

## 3.1 安全问题

- LLM / embedding 配置中存在明文 API Key。
- `/debug/ai/**` 在安全配置中匿名放行。
- 调试能力默认开启，不符合生产默认安全策略。

## 3.2 工程问题

- 无 `src/test` 自动化测试体系。
- 回归更多依赖人工调用。
- 任务状态虽有 `task_record`，但仍主要是同步执行，不是正式异步治理模式。
- 缺少结构化审计日志和关键操作日志口径。

## 3.3 错误语义问题

- 当前已有 `BusinessException` 和全局异常处理，这是基础，但还缺少更明确的业务错误口径整理。
- 部分错误虽能返回，但对前后端协作和排障来说语义仍不够标准化。

## 4. 第三周必须完成项

## 4.1 debug AI 接口收口

### 当前问题

- `/debug/ai/**` 在安全配置中被 `permitAll` 放行。
- `app.debug.ai-test.enabled=true` 在当前配置中默认启用。

### 第三周设计

- 默认环境关闭 debug AI 接口。
- 调试接口只能在显式开启的本地或联调环境使用。
- 调试接口应至少满足以下之一：
- 仅 `dev` profile 启用。
- 仅管理员可访问。
- 必须带独立调试开关。

### 推荐方案

- 配置层：默认 `app.debug.ai-test.enabled=false`
- 安全层：移除匿名放行，不再 `permitAll`
- 权限层：若保留，则要求管理员访问

## 4.2 API Key 管理设计

### 当前问题

- 真实密钥直接出现在 `application.yml` / `application-dev.yml`

### 第三周设计

- 配置文件中只保留环境变量占位符。
- 本地开发使用环境变量或外部未提交配置文件。
- 文档明确：
- 开发环境如何注入。
- 测试环境如何注入。
- 生产环境如何注入。

### 完成标准

- 仓库内不再出现真实 API Key。

## 4.3 配置隔离设计

### 目标

区分：

- 本地开发。
- 联调 / 验收环境。
- 生产环境。

### 第三周建议

- 敏感能力默认关闭，显式开启。
- mock / real 模式通过配置清晰区分。
- debug 能力与生产运行能力分离。

### 注意

当前项目同时存在真实 embedding、本地 hash embedding、历史 mock 客户端等能力，第三周必须避免“运行模式不清楚”的状态。

## 4.4 异常码与错误语义规范

### 当前现状

项目已有：

- `BusinessException`
- 全局异常处理
- 基础 HTTP 状态与业务码映射

### 第三周目标

建立更明确的异常语义分层：

- 参数错误
- 登录态错误
- 权限错误
- 资源不存在
- 状态冲突
- 外部依赖失败
- 系统异常

### 建议

为以下关键链路整理统一错误口径：

- 文档上传
- 文档解析
- embedding
- retrieval
- chat
- conversation
- admin

第三周不要求做独立错误码中心平台，但要形成文档规范和接口一致性。

## 4.5 日志与审计设计

### 当前问题

当前有基础日志，但缺少统一的关键链路日志视角。

### 第三周建议日志重点

- 文档上传结果。
- 文档处理任务开始 / 成功 / 失败。
- embedding 执行结果。
- retrieval 命中数量与有效数量。
- chat 生成结果与拒答原因。
- 管理端关键治理动作。

### 审计建议

第三周不强制做独立审计表，但应至少明确后续关键动作需要审计：

- 文档删除。
- 文档替换。
- 重新处理。
- 重新 embedding。
- 管理端治理动作。

## 4.6 测试与回归设计

### 当前现状

- 主要依赖 Apifox 手工验收。
- 没有 `src/test`。

### 第三周最小目标

- 建立最小单元测试/集成测试骨架。
- 对最关键链路建立回归覆盖：
- 认证。
- 文档上传。
- 文档处理。
- embedding 状态。
- 检索。
- 问答。
- 权限隔离。

### 第三周现实策略

不要求一次补齐全面自动化测试，但必须至少做到：

- 有可运行测试工程入口。
- 有最关键路径的自动化样例。
- 手工验收与自动化回归口径不冲突。

## 4.7 异步任务演进建议

### 当前现状

- 已有 `task_record` 和状态流转。
- 当前 parse / embed 仍以同步执行为主。

### 第三周设计目标

- 继续保留当前同步实现，避免复杂化。
- 但在设计上明确后续如何演进到轻量异步。

### 建议演进路径

第一阶段：

- 同步执行 + `task_record` 可追踪。

第二阶段：

- 单机异步执行器。

第三阶段：

- 真正消息队列或分布式任务系统。

第三周只要求把演进边界文档化，不要求上 MQ。

## 5. 第三周建议完成项

### 5.1 错误响应补强

- 问答拒答与模型不可用场景区分错误或状态语义。
- 文档生命周期动作冲突语义统一。

### 5.2 运行模式识别

- 明确当前环境到底是：
- real embedding 还是本地 hash embedding
- real LLM 还是占位 / 降级模式

第三周可通过：

- 配置只读信息接口
- 或日志明确标识

来解决“当前环境模式不清楚”的问题。

## 6. 与现有实现的兼容策略

- 不推翻当前 `BusinessException` 体系。
- 不推翻当前 `task_record` 模型。
- 安全收口优先通过配置和权限策略完成，不做大规模架构改造。
- 自动化测试从最小样例开始增量补齐。

## 7. 风险与注意事项

- 安全收口不能只改文档，必须落实到默认配置与安全策略。
- 测试建设不能只停留在“有计划”，至少要有可运行最小样例。
- 任务治理设计不能因为追求正式版架构而破坏当前稳定主链路。

## 8. 第三周完成标准

- 默认运行环境不再匿名暴露 debug AI 调试能力。
- 仓库配置中不再保留真实 API Key。
- 环境配置隔离策略清晰。
- 异常码和错误语义有明确规范。
- 最关键主链路开始建立自动化回归基线。
- 生命周期、问答、任务相关日志更适合排障和治理。

## 9. 与旧文档差异说明

第二周文档重点是“打通真实主链路并完成演示”。

第三周必须正视当前真实项目已经暴露出的工程和安全问题，不再把这些问题视为“上线前再说”的尾项，而是把它们提升为正式开发范围的一部分。
