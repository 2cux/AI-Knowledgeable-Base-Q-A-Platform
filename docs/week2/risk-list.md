# 第二周风险清单

## 1. 文档定位

本文档用于识别第二周从 MVP 后端骨架推进到真实可用闭环时的主要风险。

当前项目已经完成认证、知识库、文档元数据、文档切片、mock embedding、mock retrieval、占位问答、问答记录、反馈、未命中问题和管理端日志。第二周的核心风险不在于“缺少模块”，而在于真实能力接入后，主链路可能因为文件处理、任务状态、AI 服务、检索质量、权限隔离或 mock 代码遗留而不稳定。

风险等级说明：

- 高：可能直接阻塞第二周核心交付物。
- 中：可能影响验收质量或造成返工，但有可控降级方案。
- 低：影响局部体验或后续维护，不阻塞主链路。

当前暴露程度说明：

- 已暴露：当前代码或文档已经明确存在该问题。
- 高概率：当前实现尚未覆盖，第二周推进中大概率出现。
- 潜在：目前未直接暴露，但真实联调后可能出现。

## 3. 详细风险清单

### R01 文件上传失败或文件残留

风险描述：

真实文件上传涉及文件系统和数据库两个写入点。任何一方失败，都可能出现“文件已保存但数据库无记录”或“数据库有记录但文件不存在”。

触发条件：

- 文件保存过程中抛异常。
- document 入库失败。
- 上传过程中应用中断。
- 文件名冲突或目录权限不足。

影响范围：

- 文档上传接口。
- 文档解析。
- 后续 parse/run 和 process。
- 文件存储目录清理。

风险等级：高

当前暴露程度：高概率

缓解方案：

- 文件路径由服务端生成，避免覆盖。
- 文件保存后再入库时，入库失败要清理已保存文件。
- 入库前先校验目录可写。
- 上传成功后立即查询文件是否存在。
- 对异常写清楚日志和业务错误。

降级方案：

- 若文件清理暂未完善，必须记录残留文件风险。
- 第二周只支持 txt/md 小文件，降低文件处理复杂度。
- 暂不开放大文件和批量上传。

是否影响第二周核心交付物：是

### R02 文件解析失败导致切片链路中断

风险描述：

当前解析器只接受请求传入的 `textContent`。第二周改成真实文件解析后，文件不存在、编码异常、空内容、不支持格式都会中断切片。

触发条件：

- `storagePath` 指向的文件不存在。
- 文件为空。
- 文件编码不兼容。
- 上传了 pdf/doc/docx，但解析器只支持 txt/md。
- 文件内容过大。

影响范围：

- 文档解析。
- 文档切片。
- parse 任务状态。
- 后续 embedding 和检索。

风险等级：高

当前暴露程度：已暴露

缓解方案：

- 第二周明确 P0 只支持 txt/md 真实解析。
- 对不支持格式直接返回业务错误。
- 解析前检查文件存在和大小。
- 空内容返回明确错误。
- 解析失败时写入 task_record.errorMessage 和 document.parseStatus。

降级方案：

- 保留 `textContent` 调试入口。
- 若真实解析失败，可用 textContent 跑通后续链路，但验收结论必须标注真实解析未通过。

是否影响第二周核心交付物：是

### R03 文档状态卡在处理中

风险描述：

当前切片逻辑会先将文档状态改为 `CHUNKING`。如果中途异常，文档可能永久停留在处理中，用户无法判断真实状态。

触发条件：

- 解析异常。
- 切片异常。
- 数据库写入 chunk 异常。
- 应用在处理中重启。

影响范围：

- 文档详情。
- 文档任务查询。
- 重新解析。
- 后续 embedding。

风险等级：高

当前暴露程度：高概率

缓解方案：

- 在事务内捕获异常并设置 `FAILED`。
- 明确 `parseStatus` 枚举：`UPLOADED`、`PENDING`、`CHUNKING`、`CHUNKED`、`FAILED`。
- task_record 同步写入失败原因。
- 提供重试或 force 重新处理入口。

降级方案：

- 如果状态恢复逻辑未完善，提供手动重新处理接口。
- 保留数据库手动修复说明，但不能作为长期方案。

是否影响第二周核心交付物：是

### R04 task_record 只记录不执行

风险描述：

当前 `/api/documents/{id}/parse` 已能创建 `task_record`，但没有执行器。若第二周不补齐执行闭环，任务状态只能停留在 `PENDING`，无法验收真实处理流程。

触发条件：

- 只实现任务创建，不实现 parse/run。
- 没有任务执行服务。
- 没有状态查询接口。

影响范围：

- 文档解析任务。
- 任务状态流转。
- 验收标准。
- 管理和排障。

风险等级：高

当前暴露程度：已暴露

缓解方案：

- 第二周必须新增 `POST /api/documents/{id}/parse/run`。
- 至少实现同步执行闭环。
- 增加 `GET /api/tasks/{id}` 和 `GET /api/documents/{id}/tasks`。
- 明确任务状态流转。

降级方案：

- 如果异步任务做不完，先做同步执行。
- 不引入复杂调度器，避免扩大范围。

是否影响第二周核心交付物：是

### R05 任务并发执行导致重复 chunk 或状态覆盖

风险描述：

同一文档被多次触发解析或 embedding 时，可能出现重复删除/插入 chunk、状态互相覆盖、embedding 指向旧 chunk 等问题。

触发条件：

- 用户连续点击解析。
- 多个请求同时调用 parse/run。
- force 重建切片时旧 embedding 未处理。

影响范围：

- document_chunk。
- chunk_embedding。
- task_record。
- 检索结果。

风险等级：中

当前暴露程度：高概率

缓解方案：

- 同一文档处于 `RUNNING` 或 `CHUNKING` 时拒绝重复执行。
- 查询 pending/running 任务并复用。
- 重建 chunk 前清理旧 chunk，依赖外键级联或显式清理 embedding。
- 状态更新使用明确条件。

降级方案：

- 第二周只允许单文档单任务同步执行。
- 暂不支持并发处理同一文档。

是否影响第二周核心交付物：是

### R06 embedding 服务接入不顺

风险描述：

真实 embedding 涉及外部服务地址、认证、模型名、超时、返回格式、向量维度等。任一不稳定都会阻塞真实向量化。

触发条件：

- embedding 服务不可用。
- API key 缺失或错误。
- 模型返回格式与客户端不一致。
- 超时或限流。
- 向量维度不符合向量库要求。

影响范围：

- `/api/documents/{id}/embed`。
- chunk_embedding 状态。
- 向量索引写入。
- 检索。

风险等级：高

当前暴露程度：高概率

缓解方案：

- 保留 MockEmbeddingClient 作为 dev profile。
- 新增 real 实现时用配置开关隔离。
- 对超时、限流、认证失败做明确错误记录。
- 每个 chunk 独立标记成功/失败。
- 先用少量 chunk 验证。

降级方案：

- 如果真实 embedding 未接通，第二周保留 mock embedding，验收结论写“真实 embedding 未通过”。
- 优先保证真实文件处理和任务闭环。

是否影响第二周核心交付物：是

### R07 向量库接入或索引写入不稳定

风险描述：

真实向量检索需要向量索引、metadata 过滤、向量写入和查询。选型或接入不顺会直接影响检索和问答。

触发条件：

- 向量库未部署。
- 索引 schema 未确定。
- metadata 未包含 knowledgeBaseId。
- 写入成功但查询不到。
- 重建索引时旧数据未清理。

影响范围：

- embedding 后处理。
- retrieval/search。
- chat/ask。
- 数据隔离。

风险等级：高

当前暴露程度：高概率

缓解方案：

- 明确第二周最小向量库方案。
- metadata 必须包含 `knowledgeBaseId`、`documentId`、`chunkId`。
- 检索必须按 `knowledgeBaseId` 过滤。
- 保留 MockVectorSearchAdapter。
- 先支持新增写入，不急于做复杂更新删除。

降级方案：

- 向量库未完成时继续使用 MySQL + mock 向量排序。
- 验收中标注 retrievalMode=MOCK，不宣称真实向量检索完成。

是否影响第二周核心交付物：是

### R08 检索结果质量不稳定

风险描述：

即使接入真实 embedding 和向量库，检索结果也可能相关性差，导致 LLM 答案质量低。

触发条件：

- chunkSize/overlap 不合适。
- 文档解析质量差。
- embedding 模型不适合中文或企业文档。
- topK 设置不合理。
- 向量库 score 阈值未设计。

影响范围：

- retrieval/search。
- chat/ask。
- citations。
- 用户体验。

风险等级：中

当前暴露程度：高概率

缓解方案：

- 第二周准备固定测试文档和固定问题。
- 验收时检查 citations 是否来自相关 chunk。
- topK 限制在合理范围。
- 暂不做复杂 rerank，但保留后续优化空间。

降级方案：

- 检索质量不稳定时，先保证可召回和可追溯。
- 把 rerank、关键词混合检索、chunk 策略优化放入后续计划。

是否影响第二周核心交付物：是

### R09 LLM 调用不稳定、超时或失败

风险描述：

真实 LLM 调用依赖外部服务和网络，可能出现超时、限流、模型错误、返回格式不可控等问题。

触发条件：

- LLM 服务不可用。
- API key 错误。
- 请求超时。
- prompt 过长。
- 模型返回空内容或错误格式。

影响范围：

- chat/ask。
- chat_record。
- 用户问答体验。

风险等级：高

当前暴露程度：高概率

缓解方案：

- 保留 SimpleAnswerGeneratorServiceImpl 作为 fallback。
- 增加 answerMode 标识：`PLACEHOLDER` 或 `LLM`。
- LLM 调用设置超时。
- 未命中时走固定拒答，不调用或少调用 LLM。
- 失败时不保存伪成功记录，或明确记录降级。

降级方案：

- 如果 LLM 未接通，继续使用占位答案，但必须在验收结论中标注。
- 第二周不做流式输出，降低实现复杂度。

是否影响第二周核心交付物：是

### R10 LLM 成本不可控

风险描述：

真实 LLM 按 token 或请求计费。测试和重复调用可能造成成本超预期。

触发条件：

- 大量重复问答测试。
- prompt 携带过多 chunk 或历史记录。
- 没有 topK 限制。
- 没有最大问题长度限制。

影响范围：

- 开发成本。
- 演示稳定性。
- 后续运营成本预估。

风险等级：中

当前暴露程度：潜在

缓解方案：

- 限制 question 长度。
- 限制 topK。
- 限制历史问答条数。
- 第二周使用小文档、小样例验收。
- 增加模型配置，便于切换低成本模型。

降级方案：

- 本地开发默认 mock。
- 只有验收环境开启真实 LLM。
- 如果成本风险较高，先只接入 embedding 和 retrieval，LLM 延后。

是否影响第二周核心交付物：否，但影响真实演示质量

### R11 未命中时 LLM 编造答案

风险描述：

当检索没有命中相关 chunk 时，如果仍然调用 LLM 或 prompt 约束不足，模型可能编造答案，破坏知识库问答可信度。

触发条件：

- retrieval 返回空结果。
- matched=false 但仍拼接普通 prompt。
- prompt 未明确要求基于上下文回答。
- 模型忽略约束。

影响范围：

- chat/ask。
- chat_record.matched。
- 管理端未命中问题。
- 用户信任。

风险等级：高

当前暴露程度：潜在

缓解方案：

- 检索为空时直接返回知识不足提示。
- matched=false 时不调用 LLM，或使用固定拒答 prompt。
- chat_record 准确记录 matched=false。
- 管理端 missed 列表用于补知识。

降级方案：

- 第二周先采用固定未命中回复。
- 暂不做“模型自由回答”。

是否影响第二周核心交付物：是

### R12 权限隔离不严导致数据泄露

风险描述：

文档、chunk、embedding、检索、问答都按知识库关联。如果任一环节漏掉 owner 校验，可能跨用户访问文档或召回 chunk。

触发条件：

- 新增接口复用 Mapper 时未校验知识库 owner。
- 任务查询只按 taskId 查，不校验文档归属。
- 向量检索 metadata 未按 knowledgeBaseId 过滤。
- conversationId 查询未按 userId 限制。

影响范围：

- 文档上传、解析、任务查询。
- 检索和问答。
- 管理端与用户端边界。

风险等级：高

当前暴露程度：潜在

缓解方案：

- 所有用户端接口都从 CurrentUser 获取 userId。
- 通过 knowledgeBase.ownerId 校验归属。
- 任务查询通过 bizId 关联 document 再校验权限。
- 向量检索强制带 knowledgeBaseId filter。
- 验收中必须使用两个用户做越权测试。

降级方案：

- 如果向量库 metadata 过滤未完成，不允许开启真实向量检索。
- 保留 mock 检索且只查当前 knowledgeBaseId 下数据。

是否影响第二周核心交付物：是

### R13 管理端权限误放开

风险描述：

管理端可以查看全量问答日志和未命中问题。如果 ADMIN 校验遗漏，普通用户可能查看系统级数据。

触发条件：

- 新增管理接口未调用 AdminPermissionService。
- Security 只校验登录，不校验角色。
- 用户 role/status 数据不规范。

影响范围：

- `/api/admin/**`。
- 问答日志。
- 未命中问题。
- 后续任务管理接口。

风险等级：高

当前暴露程度：潜在

缓解方案：

- 所有管理端 Service 首行调用 `adminPermissionService.ensureAdmin()`。
- 验收普通用户访问管理端必须返回 403。
- ADMIN 用户必须 status=1。

降级方案：

- 第二周不新增管理任务接口，降低风险面。
- 如新增 `/api/admin/tasks`，必须复用同一权限服务。

是否影响第二周核心交付物：是

### R14 状态枚举分散导致状态流转混乱

风险描述：

当前文档状态、任务状态、embedding 状态分散在不同服务中用字符串常量维护。第二周状态增多后，容易出现状态不一致。

触发条件：

- 多个服务各自定义 `PENDING`、`FAILED` 等字符串。
- document.parseStatus 和 task_record.status 不同步。
- embed 状态和文档总体状态口径不一致。

影响范围：

- 文档详情。
- 任务查询。
- embedding-status。
- 验收判断。

风险等级：中

当前暴露程度：已暴露

缓解方案：

- 抽取状态常量类或枚举。
- 明确状态流转图。
- 文档、任务、embedding 分别定义边界，不混用。
- 验收文档以状态流转为标准。

降级方案：

- 第二周可以先保留字符串，但集中到常量类。
- 暂不做复杂状态机框架。

是否影响第二周核心交付物：是

### R15 mock 与 real 实现混用导致验收结论失真

风险描述：

第二周会同时存在 mock 和 real 实现。如果接口响应不标注当前模式，可能误把 mock 链路当成真实链路验收通过。

触发条件：

- Spring Bean 注入了 mock 实现。
- 配置未区分 dev/real profile。
- 响应中没有 `embeddingMode`、`retrievalMode`、`answerMode`。
- 验收人员不清楚当前环境模式。

影响范围：

- embedding。
- retrieval。
- chat/ask。
- 验收结论。

风险等级：高

当前暴露程度：高概率

缓解方案：

- 配置中明确 `ai.embedding.mode`、`ai.retrieval.mode`、`ai.answer.mode`。
- 响应或日志中输出当前模式。
- 可选增加 `/api/system/ai-capabilities`。
- 验收结论必须写明 MOCK / REAL。

降级方案：

- 如果 real 未完成，明确写“有条件通过”。
- 不允许把 mock 演示描述为真实 RAG。

是否影响第二周核心交付物：是

### R16 旧 mock 代码遗留影响真实链路

风险描述：

现有 mock embedding、mock retrieval、占位答案是骨架阶段必要实现。第二周引入真实能力后，如果没有清晰隔离，旧实现可能继续被默认注入。

触发条件：

- `@ConditionalOnMissingBean` 使用不当。
- 新增 real Bean 未生效。
- mock 实现和 real 实现同时存在导致冲突。
- 测试环境和本地环境配置不同。

影响范围：

- AI 链路。
- 回归测试。
- 演示效果。

风险等级：中

当前暴露程度：已暴露

缓解方案：

- 使用 profile 或配置条件明确选择实现。
- 启动日志打印当前实现类。
- 保留 mock，但默认环境要符合预期。
- 编写最小模式验证接口或日志。

降级方案：

- 第二周先只做配置隔离和显式标识。
- real 实现不稳定时可切回 mock，但验收结论标注。

是否影响第二周核心交付物：是

### R17 citations 序列化或解析失败

风险描述：

问答记录保存 citations_json，第二周要求用户详情和管理详情返回 citations。如果 JSON 序列化或反序列化失败，会影响答案可追溯。

触发条件：

- CitationVO 字段变更。
- 历史记录 citations_json 为空或格式异常。
- ObjectMapper 解析异常。

影响范围：

- chat/ask。
- chat_record。
- 用户问答详情。
- 管理端问答详情。

风险等级：中

当前暴露程度：潜在

缓解方案：

- 保存时统一序列化。
- 查询时解析失败返回空列表并记录日志，或返回明确错误。
- 对历史空值兼容。
- 增加最小单元测试或手动验收。

降级方案：

- 如果结构化解析不稳定，详情先返回原始 citationsJson，但标记后续优化。

是否影响第二周核心交付物：是

### R18 真实链路接入后回归破坏原有接口

风险描述：

新增文件上传、任务、real AI 实现时，可能修改现有 Service 和 VO，导致已有接口行为变化或编译失败。

触发条件：

- 修改 DocumentServiceImpl、ChatServiceImpl、RetrievalServiceImpl 时影响已有路径。
- VO 字段调整不兼容。
- Bean 注入冲突。
- Flyway 迁移变更影响旧表。

影响范围：

- 已完成的全部 MVP 骨架接口。
- 构建和启动。
- 第二周演示。

风险等级：中

当前暴露程度：潜在

缓解方案：

- 每次改动后运行 `mvn test`。
- 保留旧接口路径。
- 新增接口优先，不轻易改已有接口语义。
- 分阶段提交和回归。

降级方案：

- 若新接口影响旧接口，优先恢复旧接口稳定。
- real 能力通过配置关闭，先保证骨架链路可用。

是否影响第二周核心交付物：是

### R19 Flyway 或数据库结构不足以支撑新增状态

风险描述：

第二周可能需要新增字段，例如任务状态、索引状态、模型模式、错误信息等。如果数据库结构不足，功能只能写在内存或响应中，难以追踪。

触发条件：

- task_record 字段不足。
- chat_record 不保存 answerMode/model。
- document 缺少失败原因字段。
- chunk_embedding 缺少向量库索引所需字段。

影响范围：

- 任务查询。
- 问答详情。
- 管理日志。
- 排障。

风险等级：中

当前暴露程度：潜在

缓解方案：

- 优先复用已有字段：task_record.error_message、chunk_embedding.embedding_error、chat_record.citations_json。
- 必须新增字段时通过 Flyway 新增迁移。
- 第二周避免大规模表重构。

降级方案：

- answerMode/model 可以先从配置或响应中返回，不强制落库。
- 文档失败原因可先记录在 task_record，不新增 document 字段。

是否影响第二周核心交付物：是

### R20 验收缺少固定样例导致结果不可复现

风险描述：

AI 检索和问答结果受文档内容、模型、配置影响。如果没有固定测试文件和问题，验收结果难以复现。

触发条件：

- 每次使用不同文档。
- 没有固定问题。
- 没有记录当前 AI 模式。
- 没有保存接口调用步骤。

影响范围：

- 验收结论。
- 问题复现。
- 后续迭代回归。

风险等级：中

当前暴露程度：高概率

缓解方案：

- 准备固定 txt/md 测试文件。
- 准备固定命中问题和未命中问题。
- 编写端到端验收脚本。
- 记录环境、模型、模式和数据库版本。

降级方案：

- 如果暂未自动化，至少保留手工验收步骤和请求样例。

是否影响第二周核心交付物：否，但影响验收可信度

## 4. 第二周重点盯防风险

第二周每天应重点检查以下风险：

### 第 1 天

- R01 文件上传失败或文件残留。
- R02 文件解析失败导致切片链路中断。
- R12 权限隔离不严导致数据泄露。

检查点：

- 真实 txt/md 是否能上传。
- 文件是否真实落盘。
- document 是否正确入库。
- 用户 A 是否不能上传到用户 B 的知识库。

### 第 2 天

- R02 文件解析失败导致切片链路中断。
- R03 文档状态卡在处理中。
- R14 状态枚举分散导致状态流转混乱。

检查点：

- 不传 textContent 是否能解析文件。
- 空文件和不支持格式是否失败可追踪。
- 文档状态是否能从处理中恢复到成功或失败。

### 第 3 天

- R04 task_record 只记录不执行。
- R05 任务并发执行导致重复 chunk 或状态覆盖。

检查点：

- parse/run 是否形成执行闭环。
- task 状态是否可查询。
- 重复执行是否受控。

### 第 4 天

- R06 embedding 服务接入不顺。
- R07 向量库接入或索引写入不稳定。
- R08 检索结果质量不稳定。
- R15 mock 与 real 实现混用导致验收结论失真。

检查点：

- 当前环境到底是 MOCK 还是 REAL。
- embedding 成功和失败状态是否准确。
- 检索是否按 knowledgeBaseId 隔离。

### 第 5 天

- R09 LLM 调用不稳定、超时或失败。
- R10 LLM 成本不可控。
- R11 未命中时 LLM 编造答案。
- R17 citations 序列化或解析失败。
- R18 真实链路接入后回归破坏原有接口。

检查点：

- 命中问题是否有答案和 citations。
- 未命中问题是否拒答。
- 问答记录详情是否可追溯。
- 原有记录、反馈、管理日志是否继续可用。

## 5. 风险处置优先级

第二周如果时间不足，应按以下顺序处理风险：

1. 权限和数据泄露风险：R12、R13。
2. 文件上传和解析主链路风险：R01、R02、R03。
3. 任务执行闭环风险：R04、R05、R14。
4. mock/real 验收失真风险：R15、R16。
5. embedding 和向量检索风险：R06、R07、R08。
6. LLM 稳定性和成本风险：R09、R10、R11。
7. 记录追溯和回归风险：R17、R18、R19、R20。

理由：

- 权限和数据泄露不能降级接受。
- 文件和任务闭环是第二周最小交付。
- 真实 AI 能力如果不稳定，可以有条件降级到 mock，但必须明确标注。
- LLM 真实接入不稳定时，不能影响文件处理和检索链路验收。

## 6. 第二周风险验收结论规则

可以判定“通过”的情况：

- 真实文件上传、解析、切片、任务查询、权限隔离全部通过。
- embedding、检索、问答至少有可运行链路。
- 如果真实 AI 已接入，明确标注 real 模式。
- 如果 AI 仍为 mock，占位或 mock 状态必须明确暴露，不能假装真实能力完成。

可以判定“有条件通过”的情况：

- 真实文件处理闭环完成。
- 任务状态闭环完成。
- 记录、反馈、管理日志不回归。
- 真实 embedding、向量检索或 LLM 有一项未完成，但有明确降级方案和后续计划。

必须判定“不通过”的情况：

- 文件无法真实上传或保存。
- 文件无法从 storagePath 解析并切片。
- parse 任务无法查询状态。
- 普通用户可以访问他人知识库、文档、任务或问答记录。
- 普通用户可以访问管理端数据。
- 问答未命中时编造答案且无 matched=false 记录。
- mock 链路被当作真实链路验收。

## 7. 结论

第二周最大的风险不是“功能做不多”，而是“真实主链路没有打稳，却继续堆外围功能”。项目推进上应坚持：

- 先稳住文件上传、解析、切片、任务状态。
- 再接入真实 embedding 和向量检索。
- 最后接入真实 LLM。
- 所有 mock 和 real 能力必须显式区分。
- 权限隔离和数据追溯必须作为硬性验收条件。

只要这些风险被持续跟踪，即使第二周真实 AI 能力只能部分完成，项目也能形成清晰的有条件 MVP 交付，而不会陷入“接口很多但主链路不可验证”的状态。
