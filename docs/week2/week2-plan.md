# 第二周开发计划

## 1. 第二周新的核心目标

第二周的核心目标不是继续横向增加模块，而是把当前已经跑通的 MVP 级 RAG 后端骨架，推进到“真实可用闭环”。

当前项目已经完成认证、知识库、文档元数据、文档切片、mock embedding、mock retrieval、占位问答、问答记录、反馈、管理日志和未命中问题查询。也就是说，外围框架和主流程编排已经具备基础。第二周应优先解决主链路里的真实能力缺口：

- 文档必须从“只保存元数据”升级为“真实文件可上传、可保存”。
- 文档处理必须从“手动传 textContent 切片”升级为“从真实文件解析文本并切片”。
- 任务能力必须从“只创建 task_record”升级为“有执行和状态流转”。
- AI 能力必须从 mock embedding、mock retrieval、占位答案，逐步替换为真实 embedding、真实向量检索、真实 LLM 问答。

第二周结束时，项目至少应支持一条可演示的真实链路：

用户注册登录 -> 创建知识库 -> 上传真实 txt/md 文档 -> 系统解析和切片 -> 向量化 -> 检索 -> 生成带引用来源的答案 -> 保存问答记录 -> 管理员查看日志和未命中问题。

## 2. 第二周最重要的交付物

### 2.1 真实文件接入能力

交付内容：

- multipart 文件上传接口。
- 本地受控目录文件存储。
- 文档元数据与真实文件路径绑定。
- 文件类型、大小、后缀和基础安全校验。

验收结果：

- 用户可以上传一个真实 txt 或 md 文件。
- 后端能够保存文件，并在 document 表中记录真实 storagePath。

### 2.2 真实文档解析与切片闭环

交付内容：

- txt/md 文件读取解析。
- 解析结果复用现有 TextSplitter 进行切片。
- 文档状态从 UPLOADED/PENDING 到 CHUNKING/CHUNKED/FAILED 的可控流转。
- 解析失败原因记录。

验收结果：

- 用户不再需要手动传 textContent，也能完成文档切片。

### 2.3 后台任务执行闭环

交付内容：

- task_record 状态流转：PENDING、RUNNING、SUCCESS、FAILED。
- 文档解析任务执行服务。
- 重复任务和并发执行保护。
- 文档处理结果可查询。

验收结果：

- 创建解析任务后，系统能够执行解析和切片，并能看到任务成功或失败状态。

### 2.4 真实 AI 能力接入方案与最小实现

交付内容：

- EmbeddingClient 从 mock 抽象扩展到真实实现。
- AnswerGeneratorService 从占位答案扩展到真实 LLM 实现。
- 配置化选择 mock 或真实实现。
- 向量检索方案确定并完成最小接入。

验收结果：

- 在有模型服务和向量检索环境时，可以走真实 embedding、真实检索、真实问答。
- 在本地开发环境仍可保留 mock profile，避免开发阻塞。

### 2.5 MVP 验收脚本或接口清单

交付内容：

- 一份可手动执行的端到端验收步骤。
- 明确每一步请求、预期响应和数据库状态变化。

验收结果：

- 第二周结束时可以按固定脚本演示完整 MVP 主链路。

## 3. 需要保留的任务

这些任务虽然来自第一周或当前已有模块，但仍是 MVP 主链路的一部分，第二周需要保留并继续完善。

- 用户注册、登录、JWT 鉴权：保留，作为全部业务接口访问基础。
- 知识库创建、列表、详情：保留，作为文档和问答的归属边界。
- 文档元数据上传：保留，但要升级为真实文件上传。
- 文档切片：保留，复用现有 TextSplitter 和 document_chunk 表。
- chunk_embedding 状态表：保留，用于真实 embedding 状态回写。
- retrieval/search 接口：保留，但实现从 mock 逐步替换为真实向量检索。
- chat/ask 接口：保留，但答案生成从占位实现替换为真实 LLM。
- 问答记录、反馈、未命中问题查询：保留，作为 MVP 运营闭环。
- 管理端问答日志：保留，用于验收问答链路和问题追踪。
- Flyway 迁移、统一响应、全局异常：保留，继续作为工程基础。

## 4. 需要提前推进的任务

以下任务原本可以放在后续阶段，但由于它们直接影响 MVP 是否真实可用，第二周应提前推进。

### 4.1 真实文件上传和存储

原因：

没有真实文件接入，知识库平台无法证明能够承载企业知识。

提前推进程度：

- 第二周必须完成 txt/md 上传和本地存储。
- pdf 可作为增强目标，视时间推进。

### 4.2 真实文件解析

原因：

当前 process 依赖 textContent，只能证明切片算法能工作，不能证明文档处理链路可用。

提前推进程度：

- 第二周必须完成 txt/md 解析。
- pdf 解析作为 P1。

### 4.3 后台任务执行闭环

原因：

文档处理、embedding 都天然适合任务化。如果只创建 task_record，不执行任务，管理和排障都会断链。

提前推进程度：

- 第二周至少完成单机同步或轻量异步执行闭环。
- 不要求分布式调度。

### 4.4 真实 embedding、真实向量检索、真实 LLM 的最小接入

原因：

当前 RAG 的核心 AI 能力都是 mock。第二周如果不推进真实接入，MVP 只能停留在骨架验证。

提前推进程度：

- 至少完成接口抽象、配置项、真实实现骨架。
- 优先完成一个可运行的最小真实链路。

## 5. 需要延后或删减的任务

第二周不应投入以下任务，除非真实主链路已经完成。

- 复杂 RBAC、部门组织、菜单权限。
- 知识库共享、成员协作、细粒度文档权限。
- 审核流、发布流、知识版本管理。
- 多知识库智能路由。
- Agent、Tool Calling、工作流编排。
- 推荐问题、热门问题聚合。
- 复杂统计看板。
- WebSocket、流式输出、长任务实时进度推送。
- 多语言支持。
- 对接 Wiki、工单、CRM、IM 等外部系统。
- 微服务拆分、高可用、分布式任务调度。
- 完整用户管理后台。

这些任务不是没有价值，而是当前投入会稀释 MVP 主链路。第二周的判断标准很简单：不能直接帮助“真实文档 -> 真实问答 -> 可追溯运营闭环”的任务，默认延后。

## 6. 任务依赖关系

第二周任务存在明确依赖，不能并行乱做。

```text
真实文件上传/存储
    -> 真实文件解析
        -> 文档切片复用与状态流转
            -> 文档 embedding
                -> 向量索引写入
                    -> 真实向量检索
                        -> 真实 LLM 问答
                            -> 问答记录、引用来源、反馈、未命中问题验收
```

详细依赖说明：

- 文件解析依赖真实 storagePath。
- 切片依赖解析结果。
- embedding 依赖 chunk。
- 向量检索依赖 embedding 成功并写入可检索索引。
- LLM 问答依赖检索结果。
- 未命中问题依赖问答记录的 matched 字段。
- 管理端日志依赖问答记录完整落库。

conversation/message、知识库维护、用户管理属于增强能力，不应阻塞真实 RAG 主链路。

## 7. 按优先级拆分的任务列表

### P0：必须完成

#### P0-1 真实文件上传与本地存储

任务内容：

- 新增 multipart 上传能力。
- 保存文件到后端受控目录。
- 生成安全 storagePath。
- 保存 document 元数据。
- 校验文件大小、后缀、基础 MIME 类型。

产出物：

- 文件上传接口。
- 文件存储配置。
- 文档元数据入库逻辑更新。
- 手动验收请求示例。

#### P0-2 txt/md 文件解析

任务内容：

- 从 storagePath 读取 txt/md 文件。
- 处理编码和空内容。
- 解析失败时抛出明确业务异常。
- 解析结果交给现有切片逻辑。

产出物：

- 文件解析服务。
- txt/md parser。
- 解析失败错误信息。

#### P0-3 文档处理任务闭环

任务内容：

- 将 parse task 从“只创建记录”升级为可执行。
- 维护 task_record 状态。
- 更新 document parseStatus。
- 失败时记录 errorMessage。

产出物：

- DocumentTaskExecutor 或等价服务。
- 任务状态流转逻辑。
- 文档处理状态可查询。

#### P0-4 真实 RAG 主链路最小验收

任务内容：

- 上传真实文档。
- 解析切片。
- 执行向量化。
- 检索 chunk。
- 发起问答。
- 保存问答记录和 citations。

产出物：

- 端到端验收步骤文档。
- 一组可复用的本地测试数据。

### P1：强烈建议完成

#### P1-1 真实 embedding 接入

任务内容：

- 增加真实 EmbeddingClient 实现。
- 配置模型服务地址、模型名、超时时间。
- 失败时写入 chunk_embedding FAILED。
- 保留 mock 实现作为 fallback 或 dev profile。

产出物：

- RealEmbeddingClient。
- embedding 配置项。
- 成功和失败状态验证记录。

#### P1-2 真实向量检索最小实现

任务内容：

- 选定向量检索方案。
- 写入 chunk 向量索引。
- 检索时按 knowledgeBaseId 过滤。
- 返回 topK、score、chunk 元数据。

产出物：

- VectorSearchAdapter 真实实现。
- 向量索引写入逻辑。
- 检索验收样例。

#### P1-3 真实 LLM 问答接入

任务内容：

- 增加真实 AnswerGeneratorService 实现。
- 设计最小 prompt 模板。
- 明确未命中时的拒答策略。
- 保留 citations。

产出物：

- RealAnswerGeneratorService。
- prompt 模板。
- 成功问答和未命中问答样例。

#### P1-4 问答详情补强

任务内容：

- 用户问答详情返回 conversationId。
- 用户问答详情返回 citations。
- 管理端详情返回 citations。

产出物：

- VO 字段补充。
- 查询映射更新。

### P2：时间允许再做

#### P2-1 PDF 文本解析

任务内容：

- 接入 PDF 文本抽取库。

产出物：

#### P2-3 知识库维护

任务内容：

产出物：

## 8. 按阶段的执行计划

### 阶段一：第 1 天，校准方案与文件接入

目标：

- 明确第二周技术方案。
- 完成真实文件上传与本地存储。

任务：

- 更新 `mvp-scope` 和 `week2-plan`。
- 设计上传接口输入输出。
- 增加文件存储配置。
- 实现 multipart 上传。
- 保留原元数据上传接口或兼容迁移。

产出物：

- `docs/week2-plan.md`。
- 文件上传接口。
- 本地文件存储目录配置。
- txt/md 上传验收通过。

### 阶段二：第 2 天，真实解析与切片闭环

目标：

- 文档可以从真实文件解析文本，并进入现有切片流程。

任务：

- 实现 txt/md parser。
- 改造 DocumentProcessService，支持从 storagePath 解析。
- 保留 textContent 作为调试入口，但不再作为唯一入口。
- 增加解析失败状态处理。

产出物：

- 文件解析服务。
- 文档解析切片闭环。
- 文档状态流转验证。

### 阶段三：第 3 天，任务执行闭环

目标：

- task_record 从记录表变成可执行任务闭环。

任务：

- 增加任务执行服务。
- 实现 PENDING -> RUNNING -> SUCCESS/FAILED。
- 防止同一文档重复执行。
- 记录失败原因。
- 明确同步执行还是轻量异步执行。

产出物：

- 解析任务执行器。
- 任务状态查询或文档状态查询增强。
- 失败场景验收。

### 阶段四：第 4 天，真实 embedding 与向量检索推进

目标：

- 开始替换 mock embedding 和 mock retrieval。

任务：

- 明确 embedding 服务和向量检索方案。
- 增加真实 EmbeddingClient。
- 增加真实 VectorSearchAdapter。

- 完成一条真实 chunk 入索引和检索链路。

产出物：

- embedding 配置。
- 向量检索配置。
- 真实向量化和检索最小样例。

### 阶段五：第 5 天，真实 LLM 问答与端到端验收

目标：

- 将问答从占位拼接升级为真实模型生成。
- 完成 MVP 主链路验收。

任务：

- 增加真实 AnswerGeneratorService。
- 设计最小 prompt。
- 接入 LLM。
- 处理未命中拒答。
- 完成端到端验收脚本。
- 修正问答详情 citations 和 conversationId 返回。

产出物：

- 真实 LLM 问答实现。
- 端到端验收文档。
- 一条可演示的真实 RAG 问答链路。

## 10. 第二周结束时项目应达到的状态

第二周结束时，项目应从“RAG 后端骨架已完成”升级为“真实 MVP 主链路可演示”。

必须达到的状态：

- 用户可以注册、登录，并访问受保护接口。
- 用户可以创建知识库。
- 用户可以上传真实 txt/md 文件。
- 系统可以从真实文件解析文本。
- 系统可以将解析文本切片入库。
- 文档处理任务有明确状态流转。
- chunk 可以完成向量化。
- 问题可以基于知识库检索相关 chunk。
- 问答接口可以返回答案、命中状态和引用来源。
- 问答记录可以落库。
- 用户可以查看历史问答并提交反馈。
- 管理员可以查看问答日志和未命中问题。
- 项目可以稳定编译，核心接口有明确验收步骤。

理想达到的状态：

- embedding、向量检索、LLM 均已接入真实服务。
- mock 与真实实现可以通过配置切换。
- pdf 解析完成基础支持。
- 用户问答详情和管理端详情均返回 citations。

不要求达到的状态：

- 不要求完整复杂权限体系。
- 不要求知识库共享协作。
- 不要求完整 conversation/message 管理。
- 不要求复杂统计看板。
- 不要求企业外部系统集成。
- 不要求微服务化或分布式任务调度。

第二周的成功标准不是“新增了多少接口”，而是“真实文档能否进入系统，并通过真实或准真实 RAG 链路生成可追溯答案”。只要这条链路稳定，后续第三周再扩展知识库维护、会话模型、用户管理和运营分析才有意义。
