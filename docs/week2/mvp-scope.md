# MVP 范围说明

## 1. 文档定位

本文档用于基于当前项目的实际开发进度，重新定义 AI 知识库问答平台的 MVP 范围。

当前项目已经不是“只完成第一周规划和工程骨架”的状态，而是已经形成了一条 MVP 级 RAG 后端骨架链路：用户可以登录、创建知识库、登记文档、传入文本切片、执行 mock 向量化、进行 mock 检索、发起占位问答，并沉淀问答记录、用户反馈和管理端日志。

因此，当前 MVP 的重点不应再继续横向增加新模块，而应优先把主链路中的 mock 能力替换为可真实运行、可验收、可演示的能力。

## 2. 当前 MVP 已完成能力

### 2.1 用户与认证

- 用户注册：支持用户名、密码、确认密码、邮箱字段校验，密码使用 BCrypt 加密后入库。
- 用户登录：支持用户名密码登录，登录成功后返回 JWT。
- JWT 鉴权：注册和登录接口放行，其余业务接口要求携带有效 Bearer Token。
- 当前用户识别：业务服务可从安全上下文获取当前登录用户 ID。

完成程度：已完成基础链路。

### 2.2 知识库管理

- 创建知识库：支持当前登录用户创建知识库，并绑定 ownerId。
- 知识库分页列表：支持查询当前用户拥有的启用状态知识库。
- 知识库详情：支持按 ID 查询当前用户可访问的知识库详情。
- 数据隔离：知识库查询和后续文档、检索、问答链路均基于当前用户做归属校验。

完成程度：已完成基础链路。

### 2.3 文档管理与切片

- 文档元数据上传：支持保存文档名称、类型、大小、存储路径、所属知识库、上传用户等元数据。
- 文件类型和大小校验：当前支持 pdf、doc、docx、txt、md 类型元数据校验，并限制文件大小。
- 文档分页和详情查询：支持按当前用户可访问范围查询文档。
- 文档解析任务记录：支持创建 DOCUMENT_PARSE 类型任务记录，并更新文档解析状态为 PENDING。
- 文档切片：支持通过请求传入 textContent，将文本按 chunkSize 和 overlap 切分，并写入 document_chunk。
- 文档切片查询：支持查询指定文档下的 chunk 列表。

完成程度：文档元数据、切片、查询已完成基础链路；真实文件上传和真实解析尚未完成。

### 2.4 向量化

- 文档 chunk 向量化：支持遍历文档 chunk，创建或更新 chunk_embedding 状态。
- 向量化状态管理：支持 PENDING、PROCESSING、SUCCESS、FAILED 状态写入。
- 向量化结果查询：支持按文档汇总查询全部 chunk 的向量化状态。
- mock embedding：当前通过确定性本地算法生成 mock vector 和 vectorId。

完成程度：已完成 mock 基础链路，尚未接入真实 embedding 模型和真实向量库。

### 2.5 检索

- 检索接口：支持在指定知识库内根据问题检索相关 chunk。
- 查询向量生成：当前复用 mock embedding 生成问题向量。
- 候选 chunk 查询：从 MySQL 查询已向量化成功的 chunk。
- 相似度排序：当前在 Java 内重新生成 mock 向量，并用余弦相似度排序返回 topK。

完成程度：已完成 mock 检索基础链路，尚未接入真实向量检索。

### 2.6 问答

- 提问接口：支持用户在指定知识库下发起问题。
- RAG 主链路串联：提问时会完成知识库归属校验、检索、答案生成、引用来源组装和问答记录保存。
- 占位答案生成：当前基于问题、最近历史问答和检索 chunk 拼接生成占位答案。
- 引用来源：返回命中的 chunk、文档 ID、文档名、chunkIndex、相似度和内容片段。
- 未命中识别：当检索不到 chunk 时，问答记录会标记 matched=false。

完成程度：已完成基础链路，但答案生成仍是占位实现，尚未接入真实 LLM。

### 2.7 轻量多轮会话

- conversationId：提问时不传 conversationId 会生成新会话 ID；传入已有 conversationId 时会加载最近历史问答。
- 历史上下文：当前按 userId、knowledgeBaseId、conversationId 加载最近 5 条 chat_record 作为上下文。

完成程度：仅完成轻量多轮能力。尚未形成独立 conversation/message 模型闭环。

### 2.8 问答记录与反馈

- 用户问答记录分页：支持当前用户查询自己的问答历史。
- 用户问答记录详情：支持当前用户查询自己的单条问答详情。
- 用户反馈：支持对自己的问答记录提交 LIKE 或 DISLIKE，并限制同一用户对同一记录重复反馈。

完成程度：已完成基础链路。

### 2.9 管理端运营能力

- 管理员权限校验：管理端接口会校验当前用户是否为启用状态的 ADMIN。
- 管理端问答日志：支持管理员分页查看全量问答记录，可按知识库和命中状态过滤。
- 管理端问答详情：支持管理员查看单条问答记录详情。
- 未命中问题查询：支持管理员查看 matched=false 的问答记录，用于发现知识缺口。

完成程度：已完成基础链路。

### 2.10 基础设施

- 统一响应结构 Result。
- 分页响应结构 PageResult。
- 全局异常处理。
- 参数校验。
- MyBatis-Plus 分页配置。
- Flyway 数据库迁移。
- Swagger/OpenAPI 依赖和基础注解。
- Maven 构建已可通过。

完成程度：已完成基础工程能力。

## 3. 当前 MVP 未完成但必须近期补齐的能力

这些能力不是远期增强，而是让 MVP 从“代码链路跑通”变成“真实可演示、可验收”的必要补齐项。

### 3.1 真实文件上传与存储

当前文档上传只保存元数据，没有接收真实文件内容。

近期必须补齐：

- 支持 multipart 文件上传。
- 将文件保存到本地受控目录或对象存储。
- 记录真实 storagePath、文件大小、文件类型、原始文件名。
- 限制文件大小、文件后缀、MIME 类型。
- 避免路径穿越、重复文件名覆盖等安全问题。

验收目标：用户能上传一个 txt、md 或 pdf 文件，后端能真实保存文件，并能基于保存路径进入后续解析流程。

### 3.2 真实文件解析

当前切片依赖请求传入 textContent，没有从 storagePath 读取并解析真实文件。

近期必须补齐：

- 优先支持 txt、md 的真实读取。
- 再支持 pdf 的文本抽取。
- doc/docx 可作为下一步补齐，但需要明确错误提示和状态。
- 解析失败时写入明确失败原因。
- 解析结果进入统一切片流程。

验收目标：上传后的文档可以不传 textContent，直接触发解析和切片。

### 3.3 后台任务执行闭环

当前已有 task_record 和创建解析任务逻辑，但没有执行器。

近期必须补齐：

- 文档解析任务从 PENDING 到 RUNNING、SUCCESS、FAILED 的状态流转。
- 任务失败原因记录。
- 基础重试控制。
- 任务查询接口或复用文档状态展示。
- 避免重复并发处理同一文档。

验收目标：用户创建解析任务后，系统能异步或同步可控地完成解析、切片，并能查询处理结果。

### 3.4 真实 embedding 接入

当前 embedding 是本地 mock vector。

近期必须补齐：

- 定义真实 EmbeddingClient 实现。
- 接入一个可配置的 embedding 模型服务。
- 保存真实 vectorId 或向量库主键。
- 处理模型调用失败、超时、限流和重试。
- 保留 mock 实现作为本地开发 profile。

验收目标：chunk 能通过真实 embedding 模型生成向量，并完成状态回写。

### 3.5 真实向量检索

当前检索是在 MySQL 取候选后由 Java 计算 mock 余弦相似度。

近期必须补齐：

- 接入真实向量库或支持向量检索的存储方案。
- 写入 chunk 向量时同步写入向量索引。
- 检索时由向量检索引擎返回 topK。
- 返回 score、chunkId、documentId、documentName、content。
- 确保 knowledgeBaseId 过滤生效，避免跨知识库召回。

验收目标：问题查询能基于真实向量索引召回相关 chunk。

### 3.6 真实 LLM 问答

当前答案是占位拼接文本。

近期必须补齐：

- 定义真实 AnswerGeneratorService 实现。
- 将问题、检索上下文、历史上下文组装为可控 prompt。
- 接入 LLM 服务并返回真实答案。
- 要求答案基于检索上下文，无法回答时明确拒答或提示知识不足。
- 保留 citations，保证答案可追溯。

验收目标：用户提问后能获得由真实模型生成、且引用来源可追溯的答案。

### 3.7 conversation/message 模型闭环

当前只在 chat_record 中用 conversationId 做轻量多轮。

近期建议补齐：

- conversation 创建、列表、详情。
- message 按 USER/ASSISTANT 角色保存。
- chat_record 与 conversation/message 的关系明确。
- 支持按会话查看完整问答过程。

验收目标：用户可以看到会话列表和会话内完整消息，而不只是散列的问答记录。

### 3.8 知识库维护闭环

当前知识库只有创建、列表、详情。

近期建议补齐：

- 更新知识库名称和描述。
- 禁用或删除知识库。
- 删除知识库前处理文档、chunk、embedding、问答记录关联影响。

验收目标：用户能维护自己创建的知识库，而不是只能新增。

## 4. 当前明确不纳入 MVP 的能力

以下能力有价值，但不应进入当前 MVP 主线，否则会分散对真实 RAG 链路的投入。

- 多租户组织架构、部门树、复杂 RBAC。
- 知识库共享、协作成员、细粒度文档权限。
- 审核流、发布流、版本管理。
- 多知识库智能路由。
- Agent、Tool Calling、工作流编排。
- 自动爬取网页、连接企业 Wiki、工单、CRM、IM 等外部系统。
- 复杂数据分析看板。
- 推荐相关问题、热门问题智能聚合。
- 多语言问答。
- 流式输出、长任务进度推送、WebSocket。
- 高可用、分布式任务调度、微服务拆分。
- 内容质量评分、自动知识更新提醒。

这些能力应在真实 RAG 闭环稳定后，再按业务优先级进入后续版本。

## 5. 当前 MVP 的真实定义

当前 MVP 不应被定义为“功能菜单尽可能多”，而应被定义为：

一个单体 Spring Boot 后端，支持企业用户把真实文档接入知识库，系统能够解析文档、切片、向量化、检索，并基于检索结果调用真实 LLM 生成带引用来源的答案，同时记录问答历史、用户反馈和未命中问题，为后续知识运营提供最小数据闭环。

用一句话概括：

MVP = 真实文档接入 + 真实 RAG 问答 + 可追溯记录 + 最小运营反馈闭环。

当前代码已经完成“骨架闭环”，但还没有完成“真实能力闭环”。第二周的核心目标应是把骨架中的 mock 环节替换为真实能力，而不是继续增加外围模块。

## 6. 当前 MVP 的关键验收标准

### 6.1 认证与权限验收

- 用户可以注册和登录。
- 未登录用户不能访问业务接口。
- 用户只能访问自己的知识库、文档、问答记录。
- 管理端接口只有 ADMIN 且启用状态用户可以访问。

### 6.2 知识库与文档验收

- 用户可以创建知识库。
- 用户可以上传真实文件。
- 文件能保存到后端可控存储位置。
- 文档能从真实文件解析出文本。
- 文档能完成切片并生成 chunk。
- 用户可以查看文档、切片和处理状态。

### 6.3 向量化与检索验收

- chunk 能调用真实 embedding 服务生成向量。
- 向量能写入真实向量检索方案。
- 检索时能按指定知识库召回相关 chunk。
- topK、score、引用片段返回正确。
- 未命中时返回空结果，并能在问答记录中标记 matched=false。

### 6.4 问答验收

- 用户可以基于指定知识库提问。
- 系统能检索上下文并调用真实 LLM 生成答案。
- 答案返回引用来源。
- 检索不到内容时，系统不应编造答案，而应提示知识库未命中或知识不足。
- 支持同一 conversationId 下的基础追问。

### 6.5 记录与运营验收

- 每次问答都保存 question、answer、matched、retrievedChunkCount、topK、citations、conversationId。
- 用户可以查看自己的历史问答。
- 用户可以对问答提交 LIKE/DISLIKE 反馈。
- 管理员可以查看全量问答日志。
- 管理员可以查看未命中问题列表，用于补充知识。

### 6.6 工程质量验收

- 项目可以稳定编译。
- Flyway 迁移可以在空库上执行成功。
- 关键业务异常有明确错误码和错误信息。
- mock 实现和真实实现可以通过配置或 Spring Bean 替换，方便本地开发和演示环境切换。

## 7. 第二周为什么不应继续横向堆模块

当前项目已经具备足够多的外围能力：认证、知识库、文档元数据、切片、mock 向量化、mock 检索、问答记录、反馈、管理日志、未命中问题。继续横向增加更多模块，例如复杂权限、知识库共享、审核流、统计看板、推荐问题，会让系统看起来更完整，但不会提升 MVP 的核心可信度。

第二周最重要的问题不是“还有哪些菜单没做”，而是“这条 RAG 链路是否真的能处理一份企业文档，并给出可信答案”。

如果继续堆模块，会产生几个直接风险：

- 主链路仍停留在 mock，演示时无法证明平台真的能处理企业知识。
- 文档只能存元数据，无法验证真实知识接入。
- 检索只是 mock 相似度，无法验证真实召回质量。
- 答案只是占位拼接，无法验证问答体验。
- 后台任务没有闭环，文档处理状态无法真实流转。
- 模块越堆越多，后续替换 mock 能力时改动面会扩大。

因此，第二周应聚焦纵向打穿真实能力：

1. 真实文件上传与存储。
2. 真实文本解析，至少先支持 txt、md，随后支持 pdf。
3. 文档处理任务闭环。
4. 真实 embedding 接入。
5. 真实向量检索。
6. 真实 LLM 问答。
7. 保持问答记录、引用来源、反馈、未命中问题可追溯。

这一阶段的正确目标是把“可运行骨架”升级为“可真实演示的最小产品”。只有主链路可信，后续扩展知识库维护、权限协作、运营分析和企业系统集成才有稳定基础。

## 8. 近期开发优先级建议

### P0：打穿真实 RAG 主链路

- 文件上传与存储。
- txt/md 解析。
- 文档切片复用现有逻辑。
- 真实 embedding。
- 真实向量检索。
- 真实 LLM 答案生成。

### P1：补齐可运营闭环

- 文档处理状态查询和失败原因展示。
- conversation/message 基础模型。
- 问答详情返回 citations 和 conversationId。
- 管理端未命中问题支持补充处理标记。

### P2：补齐维护能力

- 知识库编辑、禁用、删除。
- 文档删除、重新解析、重新向量化。
- 用户状态检查和基础用户管理。

## 9. 结论

当前项目的 MVP 已经完成后端骨架，不再处于“从零搭建”阶段。下一阶段的核心不是扩大功能边界，而是把最关键的 mock 环节替换成真实能力。

只要能完成“真实文件进入系统 -> 真实解析切片 -> 真实向量化 -> 真实检索 -> 真实 LLM 生成答案 -> 记录、反馈、未命中可追踪”这一条链路，当前 MVP 就具备了对外演示和继续迭代的基础。
