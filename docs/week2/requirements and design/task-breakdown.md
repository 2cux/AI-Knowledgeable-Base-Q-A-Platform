# 第二周任务拆解

## 1. 拆解原则

第二周的任务拆解围绕一个目标：把当前已经完成的 MVP 后端骨架推进到真实可用闭环。

本周不追求新增更多业务模块，而是优先替换 mock 能力，补齐真实文件接入、真实解析、任务状态流转、真实 embedding、真实检索、真实 LLM 问答，以及对应 API 和验收文档。

优先级定义：

- P0：必须完成，直接决定第二周 MVP 主链路是否成立。
- P1：强烈建议完成，决定 MVP 是否具备真实演示价值。
- P2：时间允许再做，不阻塞第二周验收。

工作量定义：

- 小：半天内可完成。
- 中：约 1 天可完成。
- 大：1 到 2 天，或涉及多个模块联动。

AI 适配说明：

- 适合：可以让 AI 先生成代码骨架、DTO、VO、接口、文档或测试样例。
- 部分适合：AI 可生成初稿，但需要人工确认安全、配置、异常和业务边界。
- 不适合：需要人工先做方案决策、选型、密钥配置、外部服务联调。

## 2. 推荐执行顺序总览

```text
01 文档和 API 口径确认
02 文件存储配置与安全边界
03 multipart 文件上传接口
04 txt/md 文件解析
05 process 接口支持从 storagePath 解析
06 task_record 状态流转
07 parse/run 执行闭环
08 任务查询接口
09 embedding 真实实现抽象与配置
10 向量索引写入与真实检索
11 LLM 问答接入
12 问答详情和管理详情补充 citations
13 端到端验收脚本
14 P2 增强项
```

## 3. P0 任务

### T01 第二周范围、API、任务口径确认

所属模块：项目管理 / 文档

任务目标：

- 确认第二周不再横向堆模块，而是纵向打穿真实 MVP 主链路。
- 保证 `mvp-scope`、`week2-plan`、`api-list-week2`、`task-breakdown` 四份文档口径一致。

前置依赖：

- 已完成当前项目实际进度扫描。

优先级：P0

预计工作量：小

完成标准：

- `docs/week2/mvp-scope.md` 存在并说明当前 MVP 真实定义。
- `docs/week2/week2-plan.md` 存在并说明第二周目标。
- `docs/week2/api-list-week2.md` 存在并列出接口调整。
- `docs/week2/task-breakdown.md` 存在并可直接排期。

风险点：

- 文档之间口径不一致，导致后续开发任务摇摆。
- API 路径沿用旧规划而不是当前真实代码路径。

是否适合交给 AI 先生成代码/文档：

- 适合。AI 可先生成文档初稿，人工再确认优先级和边界。

推荐执行顺序：01

### T02 文件存储配置与安全边界设计

所属模块：文档 / 文件存储

任务目标：

- 为真实文件上传建立后端受控存储目录。
- 明确文件保存路径、访问限制、命名规则和安全校验策略。

前置依赖：

- T01。

优先级：P0

预计工作量：小

完成标准：

- `application.yml` 或 `application-dev.yml` 中有文件存储根目录配置。
- 文件路径由服务端生成，不直接信任客户端路径。
- 文件名做安全处理，避免路径穿越。
- 文件大小、扩展名、基础 MIME 类型有校验策略。

风险点：

- 直接使用用户上传文件名可能导致覆盖或路径穿越。
- Windows 和 Linux 路径差异导致后续部署问题。
- 文件存储目录不在受控范围，后续清理和权限管理困难。

是否适合交给 AI 先生成代码/文档：

- 部分适合。AI 可生成配置类和工具类，人工必须确认路径、安全和部署约束。

推荐执行顺序：02

### T03 新增真实文件上传接口

所属模块：文档 / API

任务目标：

- 新增 `POST /api/documents/upload-file`，支持 `multipart/form-data` 上传真实文件。
- 上传成功后保存文件，并创建 document 记录。

前置依赖：

- T02。
- 当前知识库归属校验逻辑可复用。

优先级：P0

预计工作量：中

完成标准：

- 接口接收 `knowledgeBaseId` 和 `file`。
- 校验当前用户拥有该知识库。
- 文件保存到受控目录。
- document 表写入 `fileName`、`fileType`、`fileSize`、`storagePath`、`parseStatus=UPLOADED`。
- 返回 `DocumentDetailVO` 或等价响应。
- 原 `/api/documents/upload` 保留为元数据上传，不破坏兼容。

风险点：

- multipart 配置和现有 JSON 接口并存时容易混淆。
- 文件保存成功但数据库写入失败时需要处理残留文件。
- 数据库写入成功但文件保存失败时不能产生脏数据。

是否适合交给 AI 先生成代码/文档：

- 部分适合。AI 可生成 Controller、DTO、Service 骨架，事务和文件残留处理需要人工重点审查。

推荐执行顺序：03

### T04 实现 txt/md 文件解析服务

所属模块：文档解析

任务目标：

- 从真实 `storagePath` 读取 txt/md 文件内容。
- 为后续切片提供真实文本输入。

前置依赖：

- T03。

优先级：P0

预计工作量：中

完成标准：

- 新增文件解析服务或扩展现有 `SimpleDocumentParser`。
- 支持 txt/md 文件读取。
- 空文件、文件不存在、编码异常有明确错误。
- 不传 `textContent` 时可以从 `storagePath` 解析。
- 继续保留 `textContent` 作为调试入口。

风险点：

- 文件编码不统一。
- 大文件一次性读取可能带来内存风险。
- DTO 注释和实际行为不一致需要修正。

是否适合交给 AI 先生成代码/文档：

- 适合。AI 可生成 parser 初稿和异常分支，人工确认读取限制和错误码。

推荐执行顺序：04

### T05 改造文档 process 接口支持真实文件解析

所属模块：文档处理 / 切片

任务目标：

- 改造 `POST /api/documents/{id}/process`，使其在未传 `textContent` 时从真实文件解析文本并切片。

前置依赖：

- T04。
- 现有 `TextSplitter` 和 `DocumentProcessServiceImpl`。

优先级：P0

预计工作量：中

完成标准：

- `textContent` 存在时沿用现有调试逻辑。
- `textContent` 不存在时读取并解析 `storagePath`。
- 解析成功后重建 document_chunk。
- 文档状态按 `CHUNKING -> CHUNKED` 流转。
- 解析或切片失败时状态可回写为 `FAILED` 或等价失败状态。

风险点：

- 当前失败时如果已将状态改为 `CHUNKING`，需要避免永久卡住。
- 重建 chunk 时需要考虑已有 embedding 数据是否失效。
- 真实解析失败和业务参数错误要区分。

是否适合交给 AI 先生成代码/文档：

- 部分适合。AI 可生成改造代码，但状态流转和异常事务需要人工审查。

推荐执行顺序：05

### T06 设计并实现任务状态流转规范

所属模块：任务 / 基础设施

任务目标：

- 将 `task_record` 从“只记录 pending”升级为可表达执行结果的状态流转。

前置依赖：

- 当前 `task_record` 表和 Mapper。

优先级：P0

预计工作量：中

完成标准：

- 明确任务状态：`PENDING`、`RUNNING`、`SUCCESS`、`FAILED`。
- 明确任务类型：至少支持 `DOCUMENT_PARSE`。
- 失败时写入 `errorMessage`。
- 执行时更新 `retryCount` 或保留当前值。
- 有统一任务 VO 返回任务详情。

风险点：

- 状态常量散落在多个服务中，后续难维护。
- 任务状态和 document.parseStatus 不一致。
- 重复任务并发执行导致状态覆盖。

是否适合交给 AI 先生成代码/文档：

- 适合。AI 可生成常量、VO、状态更新服务初稿。

推荐执行顺序：06

### T07 新增 parse/run 文档解析执行闭环

所属模块：文档任务 / API

任务目标：

- 新增 `POST /api/documents/{id}/parse/run`，创建并执行文档解析任务，形成第二周最小可验收闭环。

前置依赖：

- T05。
- T06。

优先级：P0

预计工作量：中

完成标准：

- 接口支持 `chunkSize`、`overlap`、`force`。
- 创建或复用解析任务。
- 执行状态从 `PENDING/RUNNING` 到 `SUCCESS/FAILED`。
- 成功时返回 `taskId`、`documentId`、`chunkCount`、`parseStatus`。
- 失败时返回或可查询 `errorMessage`。

风险点：

- 同步执行接口可能耗时较长。
- force 重建切片时需要处理旧 chunk 和旧 embedding。
- 并发调用同一个文档可能重复执行。

是否适合交给 AI 先生成代码/文档：

- 部分适合。AI 可生成接口和服务骨架，并发保护和事务边界需要人工确认。

推荐执行顺序：07

所属模块：任务 / API

是否适合交给 AI 先生成代码/文档：

- 适合。AI 可生成 Controller、Service、VO 和查询逻辑初稿。

推荐执行顺序：08

## 4. P1 任务

### T10 真实 embedding 实现与配置隔离

所属模块：Embedding

任务目标：

- 在保留 mock embedding 的前提下，新增真实 `EmbeddingClient` 实现。

前置依赖：

- T05 或已有 chunk 数据。
- 需要明确真实 embedding 服务或 API。

优先级：P1

预计工作量：大

完成标准：

- 存在 mock 和 real 两种实现。

风险点：

是否适合交给 AI 先生成代码/文档：

- 部分适合。AI 可生成接口适配骨架，外部服务联调不适合完全交给 AI。

推荐执行顺序：09

### T11 真实向量索引写入

所属模块：向量检索 / 索引

任务目标：

- 将 chunk 的真实 embedding 写入向量检索方案，形成可检索索引。

前置依赖：

- T10。
- 需要选定向量存储方案。

优先级：P1

预计工作量：大

完成标准：

- chunk 向量可写入向量索引。
- 索引元数据包含 `knowledgeBaseId`、`documentId`、`chunkId`、`chunkIndex`。
- 重复向量化时支持更新或覆盖。
- 写入失败可记录错误。

风险点：

- 向量库选型和部署成本超出第二周范围。
- 元数据过滤设计不当可能导致跨知识库召回。
- 重建 chunk 后旧索引清理不完整。

是否适合交给 AI 先生成代码/文档：

- 不适合直接生成完整实现。适合让 AI 辅助生成适配器骨架和接口注释，具体选型和联调需人工主导。

推荐执行顺序：10

### T12 真实向量检索适配器

所属模块：检索

任务目标：

- 新增真实 `VectorSearchAdapter`，让 `/api/retrieval/search` 可以走真实向量检索。

前置依赖：

- T10。
- T11。

优先级：P1

预计工作量：大

完成标准：

- 检索按 `knowledgeBaseId` 过滤。
- 返回 topK chunk。
- 返回 score、chunkId、documentId、documentName、content。
- 响应中能标识 `retrievalMode=VECTOR`。
- mock 检索仍可在 dev 环境使用。

风险点：

- 向量库 score 语义与 mock 余弦分数不一致。
- 查询向量生成失败会导致检索不可用。
- 元数据回表查询可能带来 N+1 或一致性问题。

是否适合交给 AI 先生成代码/文档：

- 部分适合。AI 可生成 adapter 框架和返回 VO 调整，真实向量库查询需人工确认。

推荐执行顺序：10

### T13 真实 LLM 问答实现

所属模块：问答 / LLM

任务目标：

- 新增真实 `AnswerGeneratorService` 实现，把占位答案替换成真实模型生成。

前置依赖：

- T12，至少需要能返回检索上下文。
- 需要明确 LLM 服务或 API。

优先级：P1

预计工作量：大

完成标准：

- 有真实 LLM 实现和 mock/placeholder 实现并存。
- 可通过配置切换。
- prompt 包含当前问题、检索 chunk、历史问答。
- 未命中时不编造答案，返回知识不足提示。
- 响应中能标识 `answerMode=LLM` 和 `model`。

风险点：

- LLM 输出不稳定，需要约束 prompt。
- 未命中策略不清会导致幻觉。
- 模型调用失败需要降级或明确报错。

是否适合交给 AI 先生成代码/文档：

- 部分适合。AI 可生成 prompt 模板和服务骨架，真实模型联调和安全策略需人工确认。

推荐执行顺序：11

风险点：

是否适合交给 AI 先生成代码/文档：

### T19 知识库维护接口

## 7. 最小可交付范围

如果第二周时间不足，必须保证以下最小交付：

- T02 文件存储配置与安全边界设计。
- T03 新增真实文件上传接口。
- T04 实现 txt/md 文件解析服务。
- T05 改造文档 process 接口支持真实文件解析。
- T06 设计并实现任务状态流转规范。
- T07 新增 parse/run 文档解析执行闭环。
- T08 新增任务查询接口。
- T09 保持现有主链路接口兼容。
- T15 端到端验收脚本。

在最小交付范围下，embedding、retrieval、answer 可以暂时保留 mock，但必须在文档和验收报告中明确标注。这种结果可以被视为“真实文件处理闭环完成，AI 能力仍待真实接入”。

## 8. 第二周完成标准

第二周结束时，项目应至少达到：

- 真实 txt/md 文件可以上传并保存。
- 真实文件可以解析并切片入库。
- 文档解析任务有状态流转和查询能力。
- 现有 mock embedding、mock retrieval、占位问答链路不被破坏。
- 如果外部服务准备完成，则支持真实 embedding、真实向量检索和真实 LLM 问答。
- 问答详情和管理详情具备引用来源追溯能力。
- 有一份可执行的端到端验收脚本。

第二周不以接口数量作为成功标准，而以“真实文档进入系统后能否形成可追踪处理链路，并尽可能接入真实 AI 能力”作为成功标准。
