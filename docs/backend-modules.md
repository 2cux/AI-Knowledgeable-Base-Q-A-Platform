# 后端模块设计（第 1 周）

## 1. 文档目标

本文档用于明确单体 Spring Boot MVP 的核心模块划分，只覆盖第 1 周需要确定的模块边界，不做微服务拆分，不引入过度抽象。

## 2. 模块划分原则

第 1 周模块划分遵循以下原则：

- 只按当前 MVP 业务闭环拆分
- 先保证职责清晰，再考虑扩展性
- 以单体 Spring Boot 为前提，不按分布式系统设计
- 对 AI 能力只预留接入位置，不展开底层实现

## 3. 核心模块列表

当前阶段模块固定为：

- `auth`
- `kb`
- `document`
- `qa`
- `conversation`
- `task`
- `config`
- `common`
- `exception`
- `infra`

## 4. 各模块职责

### 4.1 auth

职责：

- 用户注册
- 用户登录
- JWT 生成与校验
- 当前登录用户信息获取
- 基础身份认证

当前阶段不负责：

- 复杂 RBAC
- 部门级数据权限
- 多租户认证

### 4.2 kb

职责：

- 知识库创建
- 知识库列表查询
- 知识库详情查询
- 知识库基础状态管理

当前阶段不负责：

- 多知识库路由策略
- 复杂分类体系
- 知识库审核流

### 4.3 document

职责：

- 文档上传
- 文档元数据保存
- 文档列表与详情查询
- 文档状态维护
- 文档切片结果关联

当前阶段不负责：

- 全格式解析能力
- 高级文档清洗
- 文档版本管理

### 4.4 qa

职责：

- 基础提问接口
- 问答结果返回
- 引用信息结构预留
- 未命中问题记录预留

当前阶段不负责：

- Embedding 实现
- 向量检索实现
- LLM 提示工程细节
- Tool Calling

### 4.5 conversation

职责：

- 会话创建
- 消息记录
- 问答历史查询
- 会话列表与详情查询

当前阶段不负责：

- 复杂多轮上下文压缩
- 长会话记忆优化

### 4.6 task

职责：

- 文档处理任务记录
- 任务状态查询
- 失败信息记录
- 后续异步扩展点预留

当前阶段不负责：

- 复杂任务编排
- 分布式调度
- 重试策略中心化管理

### 4.7 config

职责：

- Spring Security 配置
- JWT 配置
- Swagger/OpenAPI 配置
- MyBatis-Plus 配置
- Redis 配置
- 全局基础配置

### 4.8 common

职责：

- 通用返回体
- 分页对象
- 通用常量
- 工具类
- 基础枚举

### 4.9 exception

职责：

- 业务异常定义
- 全局异常处理
- 参数校验异常处理
- 统一错误码结构

### 4.10 infra

职责：

- 封装外部基础能力接入点
- 预留 AI 模型客户端位置
- 预留 Redis / 文件存储 / MQ 适配位置

当前阶段建议：

- 只保留接口与基础封装位置
- 不深入具体模型与向量服务实现

## 5. 模块关系说明

核心依赖关系建议如下：

- `auth` 为所有需要登录的业务模块提供认证能力
- `kb` 是 `document` 和 `qa` 的上层业务归属
- `document` 为 `qa` 提供知识来源数据基础
- `conversation` 为 `qa` 提供会话和历史上下文承载
- `task` 为 `document` 提供异步处理状态记录
- `config`、`common`、`exception` 为全局支撑模块
- `infra` 为业务模块提供第三方能力接入点


## 7. 第 1 周模块设计结论

第 1 周模块设计只要求满足“登录、知识库、文档、问答、会话、任务”这条最小主链路，确保边界明确、命名统一、后续容易继续补代码即可。
